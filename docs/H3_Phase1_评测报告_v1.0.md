# H3 方案 — Phase 1 评测报告 v1.0

- **日期**: 2026-04-09
- **环境**: 单机 (2TB RAM, 4×4090D, Milvus 2.5 分布式)
- **目标**: 验证方案 H3 (DINOv2-L 1024d HNSW_SQ8) 在开发环境的端到端可行性

---

## 0. 执行摘要

| 指标 | 目标 | 实测 | 状态 |
|------|------|------|------|
| Recall@10 (近重复) | ≥ 99% | **100%** | ✅ |
| Recall@10 (压缩/UI) | ≥ 95% | **99.5-100%** | ✅ |
| Recall@10 (轻度裁剪) | ≥ 90% | **98.0%** | ✅ |
| Recall@10 (重度裁剪 30%) | ≥ 60% | **42.5%** | ❌ |
| Recall@10 (图中图) | ≥ 50% | **9.5%** | ❌ |
| **延迟 P95 (批量评测)** | < 50ms | **5.5ms** | ✅✅ |
| 延迟 P95 (单 query 服务) | < 100ms | ~500ms | ⚠️ 优化空间 |

**结论**: H3 方案对**通用场景表现极佳** (100% Recall + 5.5ms 延迟), 但对**极端裁剪和图中图场景仍是弱项**, 需要 L3 patch 重排或 YOLO 子图增强。

## 2. 6 场景 Recall 评测

### 2.1 测试集
6 场景 × 200 query = 1200 个评测查询 (从 100 万中随机抽 1000 张, 每张生成 6 个变体)

### 2.2 完整结果

| 场景 | 描述 | Recall@1 | Recall@10 | Recall@100 | P50 ms | P95 ms |
|------|------|---------|-----------|------------|--------|--------|
| 1_original | 原图 | **98.0%** | **100.0%** | 100.0% | 4.7 | 6.8 |
| 2_jpeg50 | JPEG 50% 压缩 | 95.0% | **100.0%** | 100.0% | 3.9 | 4.5 |
| 3_crop70 | 中心裁剪 70% | 81.0% | **98.0%** | 98.5% | 4.0 | 8.6 |
| 4_crop30 | 中心裁剪 30% | 20.5% | 42.5% | 64.5% | 3.9 | 4.3 |
| 5_ui_overlay | UI 叠加 | 93.0% | **99.5%** | 99.5% | 4.0 | 4.2 |
| 6_embed_large | 嵌入大图 | 3.0% | **9.5%** | 24.0% | 4.2 | 4.6 |
| **平均** | - | **65.1%** | **74.9%** | **81.1%** | **4.1** | **5.5** |

### 2.3 关键观察

**优势 (可上线生产)**:
- ✅ **原图自查询**: 100% Recall@10, 几乎完美
- ✅ **JPEG 重压缩**: DINOv2 对压缩天然鲁棒
- ✅ **UI 叠加**: 99.5% — 远超 CLIP 预期, **DINOv2 自监督训练真有用**
- ✅ **轻度裁剪 (70%)**: 98% — 符合预期

**弱点 (待 L3/YOLO 解决)**:
- ❌ **重度裁剪 (30%)**: 42.5% — 商品只有原图 30% 区域时, 全局向量已严重偏移
- ❌ **图中图 (商品嵌入网页)**: 9.5% — 商品占大图很小比例时, 全局特征被背景主导

**延迟**:
- ✅ 批量评测 P95 = 5.5ms (32 query 一批)
- ⚠️ 单 query 服务 P95 ~500ms (未来优化: 内部 batching)

## 3. 多值过滤性能 (RoaringBitmap)

### 3.1 测试矩阵

| 测试用例 | P50 | P95 | 评价 |
|---------|-----|-----|------|
| 无过滤 (基线) | 11.7ms | 14.3ms | - |
| brand_id == X | **5.8ms** | 7.2ms | ⚡ 加速 |
| brand_id in [5 个] | 6.2ms | 6.8ms | ⚡ 加速 |
| category_l1 == X | 16.1ms | 19.1ms | OK |
| brand+category 组合 | 5.8ms | 7.1ms | ⚡ 加速 |
| **array_contains(merchant_ids, 1)** | **68.0ms** | **146.2ms** | ⚠️ **慢** |
| **array_contains_any(merchant_ids, [5])** | **113.3ms** | **191.1ms** | ⚠️ **较慢** |
| 全条件 (3 字段) | 31.5ms | 84.1ms | - |

### 3.2 发现

✅ **标量字段过滤**: 5-20ms, 算子下推有效, **比无过滤更快** (因为预过滤减少了搜索空间)

⚠️ **多值字段 (ARRAY) 过滤显著慢**:
- `array_contains` 单值: 68ms (12x 基线)
- `array_contains_any` 多值: 113-191ms (10-16x 基线)

**怀疑原因**: BITMAP 索引可能未真正下推到 HNSW 遍历, 而是 post-filter

**待解决**:
1. 验证 BITMAP 索引在 ARRAY 字段上的实际效果
2. 是否需要换成 INVERTED 索引
3. 或考虑把 merchant_id 单独维护倒排表 (类似 ChatGPT 方案的外置 RoaringBitmap)

## 4. 5 分片模拟测试

### 4.1 设计
- 用 brand_id 区间模拟 5 个分片 (实际生产是物理分片)
- 5 路并行 search → Top K 合并

### 4.2 结果

| 测试 | P50 | P95 |
|------|-----|-----|
| 单 Collection 直接查询 | 12.7 ms | 17.0 ms |
| 5 分片扇出合并 | 39.1 ms | 46.8 ms |
| **扇出开销** | **+26.4 ms** | **+29.7 ms** |

### 4.3 解读

单机模拟的 30ms 扇出开销主要来自:
- ❗ **Milvus querynode 串行处理**: 同一个 querynode 处理 5 个 search 请求时不能完美并行
- Python ThreadPoolExecutor 协调

**真实 5 机环境预测**:
- 每机独立 querynode → 真并行
- 网络往返 < 5ms (内网)
- **扇出开销预计 < 10ms**

**结论**: 5 机分片可行, P95 延迟仍能控制在 30ms 以内。

## 5. 端到端搜索服务

### 5.1 服务架构
- FastAPI 服务 (`services/h3-search/main.py`)
- POST `/search` 接受 image base64 + 过滤条件
- 内部流程: decode → DINOv2 推理 → Milvus 召回 → 返回

### 5.2 性能 (单 query 测试, 5 次连续)

| # | 总延迟 | 推理 | Milvus | target_pos |
|---|--------|------|--------|-----------|
| 1 (cold) | 2206 ms | 553 ms | 1465 ms | 1 |
| 2 | 501 ms | 60 ms | 384 ms | 1 |
| 3 | 528 ms | 101 ms | 370 ms | 1 |
| 4 | 749 ms | 301 ms | 349 ms | 1 |
| 5 | 478 ms | 112 ms | 290 ms | 1 |

**正确性**: ✅ 5/5 命中 Top 1 (Recall@1=100%)

**延迟问题**:
- 单 query 比批量评测慢 (推理 60-300ms vs 批量 4ms)
- HTTP/JSON 序列化开销
- 没有内部 batching

**优化方向**: 实现 inference batching (将多个并发请求合批)

## 6. 与方案 H3 v1.1 设计的对比

### 6.1 v1.1 单机版 vs 实测

| 维度 | v1.1 设计 | **实测** | 状态 |
|------|----------|---------|------|
| 主向量维度 | 512d (PCA) | **1024d (无 PCA)** | ✅ 5机集群配置 |
| Recall@10 (通用) | 98.5% | **100%** (近重复) | ✅✅ 超预期 |
| Recall@10 (UI叠加) | 95% | **99.5%** | ✅ |
| Recall@10 (重度裁剪) | 90% | **42.5%** | ❌ 大幅低于预期 |
| 延迟 P95 (批量) | 36ms | **5.5ms** | ✅✅ 远超预期 |
| 内存 (90万图) | ~700MB | ~1GB | ✅ 符合预期 |

### 6.2 重要发现

1. **DINOv2-L 单向量足以处理通用场景** — 远超 v1.1 的 98.5% 目标
2. **延迟远低于预期** — 36ms 预算实测仅用 5.5ms (批量)
3. **极端裁剪和图中图仍是 H3 弱项** — 需要 v1.2+ 增强 (YOLO 子图 / patch 匹配)
4. **多值过滤是开发瓶颈** — RoaringBitmap 算子下推未达预期

## 8. 部署建议

### 8.1 单机生产可行性
- ✅ DINOv2-L 推理 + Milvus 单机 + 100 万级数据: **完全可上线**
- ⚠️ 单机 30 亿: 需要 256d/PCA 降级 (按 v1.1)

### 8.2 5 机集群推荐
- ✅ 用 1024d 无 PCA, Recall 最优
- ✅ 每机 6 亿数据, RAM 占用 ~1 TB (50%)
- ✅ 扇出延迟可控 (< 10ms)

### 8.3 上线路径
1. **Phase 2**: 在本机部署生产配置 (DINOv2 + 当前 1.5 亿)
2. **Phase 3**: 灰度 10% 流量, 对比新旧 Recall
3. **Phase 4**: 5 机集群扩容
4. **Phase 5**: 全量切换
