## 一、数据模型变更

### 1.1 Collection 架构: 单集合 → 5 Collection 物理隔离

**v1.2 (旧)**:
- 单一 `global_images` 集合，按 `ts_month` 分区 (18 滚动 + 1 常青 = 19 分区)
- 两区策略: 热区 HNSW (5 月滚动) + 非热区 DiskANN (13 月滚动 + 常青)

**v1.3 (新)**:
- 5 个独立 Collection，按月份和会员等级物理隔离
- 命名规则: `img_YYYYMM_vip` / `img_YYYYMM_svip` / `img_999999` (常青池)
- 物理隔离优势: 独立索引构建、独立加载/释放、故障隔离、按需扩缩

### 1.2 Collection Schema

```python
fields = [
    # === Primary Key ===
    FieldSchema("image_pk",      DataType.VARCHAR, max_length=32, is_primary=True),

    # === 向量字段 ===
    FieldSchema("global_vec",    DataType.FLOAT_VECTOR, dim=256),  # 768d→256d 投影

    # === 标量字段 (INVERTED 索引) ===
    FieldSchema("category_l1",   DataType.INT32),          # L1 品类 (18类)
    FieldSchema("category_l2",   DataType.INT32),          # L2 品类 (84类)
    FieldSchema("category_l3",   DataType.INT32),          # L3 品类
    FieldSchema("tags",          DataType.ARRAY,           # 标签数组
                element_type=DataType.INT32, max_capacity=32),
    FieldSchema("color_code",    DataType.INT32),          # 颜色编码
    FieldSchema("material_code", DataType.INT32),          # 材质编码
    FieldSchema("style_code",    DataType.INT32),          # 风格编码
    FieldSchema("season_code",   DataType.INT32),          # 季节编码

    # === 元数据字段 ===
    FieldSchema("is_evergreen",  DataType.BOOL),           # 常青标记
    FieldSchema("ts_month",      DataType.INT64),          # 写入月份 YYYYMM ★ INT32→INT64
    FieldSchema("promoted_at",   DataType.INT64),          # 常青晋升时间戳 (Unix ms)
    FieldSchema("created_at",    DataType.INT64),          # Unix 毫秒时间戳
    FieldSchema("product_id",    DataType.VARCHAR, max_length=64),  # 商品ID透传
]
```

**与 v1.2 的字段差异**:

| 字段 | v1.2 | v1.3 | 变更说明 |
|------|------|------|---------|
| `ts_month` | INT32 | **INT64** | 类型扩展，兼容更大范围 |
| 其他字段 | — | — | Schema 结构不变，字段语义不变 |

### 1.3 索引配置

**v1.2**: 热区 HNSW(M=24, efConstruction=200) + 非热区 DiskANN(MaxDegree=64)，双区策略

**v1.3**: 统一 HNSW 索引 (多 Collection 隔离后无需 DiskANN 分区策略)

```python
    "category_l3":   {"index_type": "INVERTED"},
    "tags":          {"index_type": "INVERTED"},
    "color_code":    {"index_type": "INVERTED"},
    "material_code": {"index_type": "INVERTED"},
    "style_code":    {"index_type": "INVERTED"},
    "season_code":   {"index_type": "INVERTED"},
    "is_evergreen":  {"index_type": "INVERTED"},
    "ts_month":      {"index_type": "INVERTED"},
}
```

### 1.4 PG uri_dedup 表变更

**v1.2**: 仅存储 `image_pk`, `uri_hash`, `ts_month`, `created_at`

**v1.3**: 新增 `uri` 字段 + 衍生追溯机制

```sql
CREATE TABLE uri_dedup (
    image_pk    CHAR(32)     PRIMARY KEY,
    uri_hash    CHAR(64)     NOT NULL,
    uri         TEXT         NOT NULL,          -- ★ 新增: 保留原始URI
    ts_month    INT          NOT NULL,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

**衍生追溯**: 对于衍生图片 (augmentation/variant)，`uri` 字段使用特殊协议:
```
synthetic://原始PK/variant
```
- 示例: `synthetic://a1b2c3d4e5f67890.../horizontal_flip`
- 用途: 搜索结果中可追溯衍生图片的原始来源，展示原图URL

## 二、搜索 UI 设计 (search_ui.py)

### 2.1 三种图片输入方式

| 输入方式 | 实现方案 | 说明 |
|---------|---------|------|
| **本地上传** | 前端直接 resize 224x224，base64 编码传输 | 最简路径，小图直传 |
| **COS 上传** | JS SDK + STS 临时密钥上传，服务端缩图 | 大图/批量场景，避免前端卡顿 |
| **URL 输入** | 前端 fetch 图片 + base64 编码 | 粘贴网络图片URL |

### 2.2 多图批量搜索

- 文件多选 (input multiple)
- 逐张异步搜索，第一张返回结果即刻显示
- 缩略图栏展示所有已搜索图片，点击切换结果
- 搜索进度指示: 已完成/总数

### 2.3 结果展示

- **图片网格布局**: 响应式瀑布流
- **信息叠加**: 每张结果图显示:
  - `L1品类/L2品类` 标签
  - 相似度分数 (0.00-1.00)
  - 品类加权标识 (同品类匹配时显示加权图标)

### 2.4 品类加权排序

搜索结果的最终排序引入品类加权:

```python
# 加权公式
if query_l1 == result_l1:
    weighted_score = raw_score + 0.05   # 同L1 +0.05
if query_l2 == result_l2:
    weighted_score = raw_score + 0.05 + 0.03  # 同L2 额外 +0.03
# 最终按 weighted_score 降序排列
```

## 三、搜索流程设计

### 3.1 完整搜索链路

```
图片输入
  ↓
resize 224x224 → base64 编码
  ↓
inference-service (GPU)
  ├── ViT-L-14: 提取 768d 特征 → 投影矩阵(seed=42) → 256d global_vec
  └── FashionSigLIP: 预测 L2/L3 品类
  ↓
Milvus 两路搜索 (并发)
  ├── 路线A: 全局搜索 top-20 (无过滤)
  └── 路线B: 同L1品类过滤搜索 top-20 (expr: category_l1 == query_l1)
  ↓
合并去重 + 品类加权排序
  ↓
阈值过滤: raw_score < 0.80 丢弃
  ↓
结果PK → PG 查 URI (支持衍生→原始追溯)
  ↓
返回结果
```

### 3.2 与 v1.2 搜索流程的主要差异

| 环节 | v1.2 | v1.3 |
|------|------|------|
| 特征提取 | TensorRT 推理 (backbone_fp16.engine) | ViT-L-14 768d → 256d 投影 + FashionSigLIP L2/L3 |
| 向量维度 | 256d (直出) | 768d → 256d (投影矩阵压缩) |
| 搜索策略 | 热区+非热区级联 + 子图+标签多路召回 | 两路并发: 全局 top-20 + 同L1过滤 top-20 |
| 排序 | Refine 精排 + 融合排序 | 品类加权排序 (同L1 +0.05, 同L2 +0.03) |
| 阈值 | 无硬阈值 (由 Refine 控制) | raw_score < 0.80 不返回 |
| Collection 加载 | 搜索时按需 load/release | **启动时预加载，搜索时不 load/release** |
| 结果URL | 直接返回 | 支持衍生→原始追溯 (synthetic://origPK/variant) |

### 3.3 投影矩阵规格

```python
# 768d → 256d 降维投影
# 生成方式: numpy.random.RandomState(seed=42) + SVD 分解
# 保证可复现: 任何环境使用相同 seed=42 可重建完全一致的投影矩阵
```

### 3.4 Collection 预加载策略

- 所有活跃 Collection 在服务启动时执行 `collection.load()`
- 搜索请求中 **不执行** `load()` / `release()`，避免加载抖动
- 非活跃 Collection (过期月份) 通过运维脚本定期 release

## 四、搜索排序需求 (待实现)

### 4.1 四级优先级排序

| 优先级 | 名称 | 条件 | 说明 |
|--------|------|------|------|
| P1 | 原图高度相似 | score > 0.95 | 近乎完全匹配，可能是同一张图或极微小变化 |
| P2 | 型号一致+语义一致 | score > 0.90 且同L2品类 | 同款不同角度/背景 |
| P3 | 系列型号+外观相同 | score > 0.85 且同L2品类且标签匹配 | 同系列产品，外观高度相似 |
| P4 | 颜色材质一致+外形相似 | score > 0.80 且同L1品类 | 相似品类，颜色材质接近 |

### 4.2 排序实现思路

```python
def rank_results(query_meta, results):
    ranked = []
    for r in results:
        if r.score > 0.95:
            r.priority = 1
        elif r.score > 0.90 and r.category_l2 == query_meta.category_l2:
            r.priority = 2
        elif r.score > 0.85 and r.category_l2 == query_meta.category_l2 and tags_match(r, query_meta):
            r.priority = 3
        elif r.score > 0.80 and r.category_l1 == query_meta.category_l1:
            r.priority = 4
        else:
            continue  # 不满足任何优先级，丢弃
        ranked.append(r)
    return sorted(ranked, key=lambda x: (x.priority, -x.score))
```

> **状态**: 待实现。当前版本使用简化的品类加权排序 (见 §二)，完整四级排序将在搜索精度提升后落地。

## 五、品类分类模型

### 5.1 两级分类架构

**v1.2**: 单模型分类 (TensorRT backbone)

**v1.3**: 两级级联分类

```
Stage 1: ViT-L-14 零样本分类
  → 输出: L1 品类 (18类)
  → 类别: 服装/箱包/鞋帽/珠宝/手表/眼镜/配饰/美妆/家居/家具/
          食品/数码/运动/汽配/宠物/母婴/文具/其他

Stage 2 (分支):
  ├── 时尚类 (服装/箱包/鞋帽等): FashionSigLIP → L2/L3 细粒度品类
  └── 非时尚类: ViT-L-14 零样本 → L2 品类
```

### 5.2 品类规模

- **L1**: 18 个大类
- **L2**: 84 个品类 (映射表维护在 `taxonomy.py`)
- **L3**: 由 FashionSigLIP 动态输出

### 5.3 投影矩阵参数

- 输入维度: 768 (ViT-L-14 输出)
- 输出维度: 256 (Milvus 存储)
- 生成方式: `numpy.random.RandomState(seed=42)` 生成随机矩阵 + SVD 分解取前 256 个奇异向量
- 可复现性: seed 固定，任何环境生成结果一致

## 六、搜索精度基线 (测试数据)

### 6.1 当前测试结果

| 指标 | 数值 | 测试条件 |
|------|------|---------|
| L1 品类准确率 | **100%** | 18 类零样本分类 |
| Top-5 精准率 | **40%** | 短袖T恤测试集 |
| Top-8 精准率 | **37.5%** | 短袖T恤测试集 |

### 6.2 精度瓶颈分析

**核心问题**: 256 维向量无法区分细粒度属性差异

- 短袖 vs 长袖 T恤: 向量距离过近，无法有效区分
- 同品类不同款式: 在 256d 空间中投影后丢失细节特征
- 颜色/材质差异: 投影压缩后信息损失

**改进方向** (后续迭代):
- 提升向量维度 (256d → 512d)，需评估存储和性能影响
- 引入细粒度属性标签辅助过滤 (标签匹配 + 向量搜索联合)
- Stage 2 模型精调 (FashionSigLIP fine-tune)
