# 以图搜商品系统 — 技术架构文档 v2.0 变更说明

> 版本: v2.0
> 日期: 2026-04-08
> 基线: v1.4 (2026-03-31)
> 变更周期: 2026-04-05 ~ 2026-04-08
> 版本号: 0.3.0.0 / 分支 isv3-0401

---

## 变更概述

v2.0 是从规划架构到实际落地的里程碑版本，核心变更:

1. **数据架构**: 单 Collection 分区方案 → 多 Collection 物理隔离，116M 向量记录已完成迁移
2. **Milvus 架构**: standalone 单节点 → 14 组件分布式集群，索引构建速度 4x 提升
3. **GPU 推理**: 单 GPU → 3 GPU 并发推理，两级品类分类模型上线
4. **搜索系统**: search-ui 前端直传上线，品类加权 + 相似度阈值搜索策略落地
5. **K8s 规划**: K3s 4 节点集群方案完成，20 个部署文件就绪

## 一、数据架构变更 (原 §4.4 / v1.4 §三 重写)

### 1.1 v1.4 → v2.0 数据架构对比

| | v1.4 (规划) | **v2.0 (实际落地)** |
|---|---|---|
| Collection 策略 | 单 Collection `global_images` 38 个分区 | **5 个独立 Collection 物理隔离** |
| 隔离粒度 | Partition 级别 | **Collection 级别** |
| 数据总量 | 规划 ~30 亿 | **实际 116M 向量记录 (已入库)** |
| 索引配置 | 分区级热冷索引 | **每 Collection 11 索引 (1 HNSW + 10 INVERTED)** |
| SVIP 隔离 | 共享 Collection 内 SVIP 分区 | **独立 SVIP Collection** |

### 1.2 新 Collection 布局

```
Milvus 多 Collection 物理隔离:

├── img_999999          常青 Collection      10M 向量
├── img_202604_vip      2026-04 VIP         22.8M 向量
├── img_202604_svip     2026-04 SVIP        20M 向量
├── img_202603_vip      2026-03 VIP         20.2M 向量
├── img_202603_svip     2026-03 SVIP        42.5M 向量

总计: 5 个 Collection, 116M 向量记录
每个 Collection: 1 HNSW 向量索引 + 10 INVERTED 标量索引
```

### 1.3 变更原因

| 因素 | 说明 |
|------|------|
| 运维独立性 | Collection 级 drop/rebuild 不影响其他数据 |
| 索引构建 | 各 Collection 独立构建，互不阻塞 |
| 内存管理 | 按 Collection 粒度 load/release，更精细 |
| 月度清理 | 直接 drop Collection，比 drop partition 更彻底 |

### 1.4 命名规则

```
img_{YYYYMM}_{tier}     月度 Collection (tier = vip | svip)
img_999999              常青 Collection (永不过期)
```

## 二、Milvus 架构升级 (原 §3 基础设施层)

### 2.1 v1.4 → v2.0 架构对比

| | v1.4 | **v2.0** |
|---|---|---|
| 部署模式 | standalone (64GB) | **分布式集群 (14 组件)** |
| MQ | 内嵌 RocksMQ | **Kafka (kafka:9092 PLAINTEXT)** |
| etcd | 单节点 | **3 节点集群 (已有)** |
| 索引构建 | 单 slot 串行 | **4 IndexNode 并行** |

### 2.2 分布式集群组件

```
Milvus 分布式集群 (14 组件):

协调层 (4 Coord):
├── RootCoord       全局元数据协调
├── DataCoord       数据分配与 compaction 调度
├── QueryCoord      查询节点负载均衡
├── IndexCoord      索引任务分发

接入层:
├── Proxy × 1       客户端连接入口 (19530)

数据层:
├── DataNode × 2    数据写入与持久化

查询层:
├── QueryNode × 3   查询执行 (128GB 内存)

索引层:
├── IndexNode × 4   索引构建 (并行加速)

依赖组件:
├── Kafka           消息队列 (kafka:9092 PLAINTEXT listener)
├── etcd × 3        元数据存储 (已有集群)
├── MinIO / S3      对象存储

关键配置:
  MQ_TYPE=kafka
  COMMON_SESSIONTTL=120
```

### 2.3 性能对比

| 指标 | standalone | **分布式** |
|------|-----------|----------|
| 索引构建 (单 Collection) | ~40 min | **~10 min (4x)** |
| 并发索引构建 | 1 slot 串行 | **4 IndexNode 并行** |
| 查询内存 | 单节点 64GB | **3 QueryNode × 128GB** |
| 写入吞吐 | 单 DataNode | **2 DataNode** |

## 三、GPU 推理架构变更 (原 §5.1 特征提取)

### 3.1 v1.4 → v2.0 推理架构对比

| | v1.4 | **v2.0** |
|---|---|---|
| GPU 数量 | 单 GPU inference-service | **3 GPU 并发 (GPU 0/1/3)** |
| 服务实例 | 1 个 | **3 个 (8090/8091/8092)** |
| 模型 | ViT-L-14 (768d) | **ViT-L-14 (768→256 投影) + FashionSigLIP (L2/L3 品类)** |
| 容器 | 各自独立 | **统一 gpu_worker 容器 (172.21.0.8)** |

### 3.2 推理服务部署

```
gpu_worker 容器 (172.21.0.8):

├── inference-service:8090    GPU 0    ViT-L-14 + FashionSigLIP
├── inference-service:8091    GPU 1    ViT-L-14 + FashionSigLIP
├── inference-service:8092    GPU 3    ViT-L-14 + FashionSigLIP

模型架构:
  ViT-L-14:       768 维 → 256 维投影层 → COSINE 搜索向量
  FashionSigLIP:  L2/L3 品类分类 (两级分类)
```

### 3.3 gpu_worker 容器网络

gpu_worker 通过 socat 端口转发访问外部服务:

| 服务 | 转发端口 | 目标 |
|------|---------|------|
| Milvus | 19530 | imgsrch-milvus:19530 |
| PostgreSQL | 5432 | imgsrch-postgres:5432 |
| Kafka | 9092 | imgsrch-kafka:9092 |
| Redis | 6379 | imgsrch-redis:6379 |

或通过 Docker 网络直接使用容器名访问 (imgsrch-milvus 等)。

## 四、搜索架构变更 (原 §5.1 / v1.4 §四)

### 4.1 搜索前端

```
search-ui (端口 9810):
  框架: FastAPI + 前端直传
  功能: 本地上传 / COS 上传 / URL 搜索 / 多图批量
```

### 4.2 搜索流程

```
图片输入 (上传/URL/COS)
    │
    ▼
GPU 推理 (8090/8091/8092 负载均衡)
    ├── ViT-L-14 → 256 维向量
    └── FashionSigLIP → L1/L2/L3 品类标签
    │
    ▼
Milvus COSINE 搜索 (多 Collection 并行)
    │
    ▼
品类加权重排序
    ├── 同 L1 品类: +0.05
    └── 同 L2 品类: +0.03
    │
    ▼
相似度阈值过滤 (raw_score >= 0.80)
    │
    ▼
PG 查询商品 URL / 元数据
    │
    ▼
返回搜索结果
```

### 4.3 搜索参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 向量维度 | 256 | ViT-L-14 投影后 |
| 相似度度量 | COSINE | Milvus metric_type |
| 相似度阈值 | raw_score >= 0.80 | 低于阈值不返回 |
| L1 品类加权 | +0.05 | 同一级品类加分 |
| L2 品类加权 | +0.03 | 同二级品类加分 |

### 4.4 v1.4 匹配层级 → v2.0 品类加权

| | v1.4 | **v2.0** |
|---|---|---|
| 排序策略 | P0/P1/P2 匹配层级 (0.90/0.70/0.50) | **COSINE 分数 + 品类加权** |
| 品类信息 | 未使用 | **FashionSigLIP L1/L2/L3 品类标签参与重排序** |
| 阈值 | P2 低至 0.50 | **统一阈值 raw_score >= 0.80** |

## 七、变更影响评估

### 7.1 与 v1.4 规划的偏差

| v1.4 规划 | v2.0 实际 | 原因 |
|-----------|----------|------|
| 单 Collection 38 分区 | 多 Collection 物理隔离 | 运维简单，drop Collection 更彻底 |
| standalone + 分区级索引 | 分布式 14 组件 | standalone 单 slot 索引慢，无法满足 116M 数据规模 |
| 自适应搜索 (暴力搜索/ANN) | 统一 COSINE + 品类加权 | 品类模型上线后搜索策略简化 |
| P0/P1/P2 匹配层级 | 品类加权 + 0.80 阈值 | FashionSigLIP 品类信息更有效 |

### 7.2 已完成工作

| 工作项 | 状态 |
|--------|------|
| 多 Collection 迁移 (116M 记录) | 已完成 |
| Milvus 分布式集群 (14 组件) | 已完成 |
| 3 GPU 并发推理部署 | 已完成 |
| ViT-L-14 + FashionSigLIP 模型上线 | 已完成 |
| search-ui 搜索前端 | 已完成 |
| K8s 部署文件 (20 个) | 已就绪 |
| K3s 集群实际部署 | 待执行 |

### 7.3 后续计划

| 工作项 | 优先级 | 说明 |
|--------|--------|------|
| K3s 集群部署 | 高 | 4 节点集群搭建 |
| Docker → K8s 迁移 | 高 | 服务逐步迁移 |
| 新月度 Collection 自动创建 | 中 | 每月 1 日自动创建 img_YYYYMM_vip/svip |
| 过期 Collection 自动清理 | 中 | 超过 18 月自动 drop |
| 云端灾备部署 (v1.4 §十二) | 低 | 待 K8s 稳定后规划 |
