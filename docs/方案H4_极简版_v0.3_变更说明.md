## v0.2 → v0.3 新增变更 (Gemini + Kimi)

### 变更 6: HNSW M=48 → M=24 (Kimi 提出)

**问题**: M=48 在 30 亿规模下邻接表 1.15 TB, 内存压力过大。

```
v0.2: M=48, efConstruction=1000
v0.3: M=24, efConstruction=200

30 亿 × 双向量:
  v0.2 邻接: 2 × 48 × 4B × 2 × 30亿 = 2.3 TB
  v0.3 邻接: 2 × 24 × 4B × 2 × 30亿 = 1.15 TB
  节省: 1.15 TB / 8机 = 144 GB/机

精度影响: R@10 下降 ~1%, 由 refine=FP16 弥补
```

### 变更 7: 新增 scene_type + quality_score 字段 (Kimi 提出)

```python
# 新增字段
FieldSchema(name="scene_type", dtype=DataType.INT32)
#  0=未知, 1=白底图, 2=模特图, 3=场景图, 4=截图/网页, 5=拼图

FieldSchema(name="quality_score", dtype=DataType.INT32)
#  0-100 分, CLIP/简单算法评估
#  用途: 过滤模糊/纯色/极低质图, 提升搜索结果质量
```

提取方式:
- quality_score: 图片清晰度 (拉普拉斯方差) + 信息量 (熵), CPU 极快
- scene_type: CLIP 零样本 ("a product on white background" / "a model wearing" / "a screenshot")

### 变更 8: 标签置信度阈值 (Kimi + GPT)

```python
# CLIP 标签提取时, 记录置信度
confidence = clip_similarity_score

if confidence >= 0.85:
    # 强标签: 可参与硬过滤
    tag_status = "confirmed"
elif confidence >= 0.6:
    # 弱标签: 仅参与 soft boost, 不参与硬过滤
    tag_status = "tentative"  
else:
    # 未知: 标记为 0 (未分类)
    tag_status = "unknown"
    tag_value = 0  # 不写入具体值

# 搜索时:
# WHERE category_l1 == X  → 只匹配 confirmed 的
# unknown 的图不会被硬过滤排除
```

### 变更 9: 标签 fallback 全库扫描 (Kimi 提出)

```python
# 搜索模式: filtered_instance
results = milvus.search(
    sscd_vec, 
    expr="category_l2 == T恤 AND color_code == 1",
    limit=10
)

# 如果标签过滤后结果不足 (<5 条)
if len(results) < 5:
    # fallback: 去掉标签, 全库 SSCD 扫描
    results = milvus.search(sscd_vec, limit=10)
    # 牺牲延迟 (~20ms → ~50ms) 保证召回
```

### 变更 10: 数据生命周期 (Kimi 提出)

```
商品下架:
  1. 业务系统发消息 → 标记 image_pk 为待删除
  2. Milvus delete(expr="image_pk == 'xxx'")  (软删除)
  3. 定期 compact (每周/每月): 物理回收碎片空间

图片替换:
  1. 业务系统发消息 → 新图 URL
  2. 删除旧向量 + 标签
  3. 重新推理 SSCD + CLIP → 插入新记录
  4. image_pk 不变 (保持业务关联)

定期维护:
  - 每周: Milvus compact (回收软删除空间)
  - 每月: 索引重建 (优化 HNSW 图结构)
  - 监控: 碎片率 > 30% 触发告警
```

### 变更 12: 搜索缓存层 (Kimi 提出)

```
热门 Query 缓存:
  Redis TTL 1h
  Key: hash(sscd_vec[:32])  # 向量前 32 维作为缓存 key
  Value: Top 10 结果列表
  
命中率预期: 20-30% (电商搜索有明显热点)
效果: 缓存命中时延迟 <2ms (vs Milvus ~15ms)
```

### 变更 13: CPU 降级方案预留 (Kimi 提出)

```
正常: GPU 推理 SSCD (2ms/图)
GPU 故障: SSCD ONNX Runtime on CPU (50-100ms/图)

预留:
  1. 导出 SSCD → ONNX 格式
  2. 搜索服务检测 GPU 不可用时自动切换 CPU
  3. 延迟从 ~20ms 上升到 ~120ms (仍可用)
  4. 入库暂停 (等 GPU 恢复)
```
