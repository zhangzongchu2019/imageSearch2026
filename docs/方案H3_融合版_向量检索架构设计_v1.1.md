## 2. 方案 H3+ 架构 (v1.1)

### 2.1 总览

```
┌─────────────────────────────────────────────────────────┐
│  L0: 指纹去重                                             │
│    技术: pHash + dHash (Milvus BIN_FLAT + Hamming)      │
│    存储: 16 B/图                                         │
│    延迟: ~1 ms                                           │
│    场景: 完全重复, 近重复 (压缩/轻度裁剪)                  │
│    Recall@K: ~100% (对目标场景)                          │
└─────────────────────────────────────────────────────────┘
                          ↓ 未命中
┌─────────────────────────────────────────────────────────┐
│  L1: 主召回 (DINOv2 + HNSW_SQ8 + Whitening + Bitmap)    │
│    模型: DINOv2-L 预训练 → PCA Whitening 512d (零训练)   │
│    索引: Milvus HNSW_SQ8 (M=48, efC=1000)                │
│         + refine=FP16 (内置精排)                          │
│    过滤: brand/category/merchant_id 算子下推              │
│    存储: 512 B (SQ8) + 230 B (HNSW 邻接, M=48)            │
│         = ~742 B/图                                      │
│    查询: ef_search=300, Top 300 候选                     │
│    延迟: ~25 ms                                          │
│    Recall@300: ~99.5%                                    │
│    Recall@10 (无精排): ~95%                              │
└─────────────────────────────────────────────────────────┘
                          ↓ Top 300
┌─────────────────────────────────────────────────────────┐
│  L2: 热数据精排 (扩大候选数到 300)                        │
│    对象: 近 3 个月热数据 (~2 亿)                          │
│    存储: DINOv2 512d fp16 (与主索引同维)                 │
│    计算: Top 300 → 精确余弦 → Top 10                     │
│    存储成本: 1KB × 2亿 = 200 GB                          │
│    延迟: ~8 ms (Top 300 GPU 批量余弦)                    │
│    Recall@10 (精排后): **~98.5%** ← 目标达成             │
└─────────────────────────────────────────────────────────┘
                          ↓ Top 10
┌─────────────────────────────────────────────────────────┐
│  L3: 冷数据层 (28 亿)                                     │
│    索引: DiskANN (向量在 SSD, 图结构在 RAM)               │
│    触发: 用户指定历史数据或全量搜索                        │
│    延迟: 30-50 ms                                        │
│    Recall@10: ~97%                                       │
└─────────────────────────────────────────────────────────┘
                          ↓
                  Top 10 返回结果

总延迟预算 (v1.1):
  热查询 (L0 + L1 + L2):        ~35 ms  ← 90% 场景
  冷查询 (L0 + L1 + L3):        ~55 ms  ← 10% 场景
  局部场景 (+ optional patch):  +20 ms  ← 未来扩展

精度指标 (v1.1):
  L1 Recall@300:     99.5%  ← 召回能力上限
  L1 Recall@100:     98.5%
  L1 + refine@10:    95%
  L2 Final Recall@10: **98.5%** ← 最终用户体验
  L2 Final Recall@1:  ~90%
```

### 2.2 设计原则 (v1.1 更新)

1. **零训练成本**: 使用 DINOv2 预训练权重 + 后置 PCA + Whitening, 不做 fine-tune / QAT
2. **两级向量存储**: SQ8 主索引 (压缩) + FP16 复排向量 (精度)
3. **多通道检索**: 指纹走指纹索引, 向量走向量索引, 过滤走 Bitmap
4. **冷热分层**: 热数据全内存 HNSW, 冷数据 SSD DiskANN
5. **工程可预测**: 每一步延迟 <10ms, 总延迟可预估
6. **降级可控**: 可按资源选择 256d/384d/512d 维度
7. **(v1.1 新增) 召回优先**: L1 Recall@300 是系统精度上限, 优先保障
8. **(v1.1 新增) 精排 buffer**: L2 精排候选数 = Top 300, 远大于 Top 10, 留足重排空间

## 3. 关键技术决策

### 3.1 模型选型

| 候选 | 维度 | 优势 | 劣势 | 决策 |
|------|------|------|------|------|
| CLIP ViT-L-14 (现有) | 768 | 零样本分类强, 已有数据 | **截图场景弱** | ❌ 不足 |
| SSCD ResNet50 | 512 | **副本检测 SOTA** | 无语义理解 | ⚠️ 辅助 |
| **DINOv2-L** | **1024** | **副本+语义兼顾** | 需降维 | ✅ **主力** |
| SigLIP 2 | 768 | 最新, 语义+定位 | Milvus 生态不成熟 | ⏭️ 未来 |
| EVA-CLIP | 768 | CLIP 增强 | 副本检测仍偏弱 | ❌ |

**理由**: DINOv2 自监督训练天然对裁剪/截图鲁棒 (同 SSCD), 同时保留了语义相似搜索能力 (类似 CLIP)。是唯一"单模型多场景"的候选。

### 3.2 维度选择 (v1.1 更新)

| 维度 | 单图 (SQ8) | 30亿存储 | Recall@10 | RAM 总需 |
|------|-----------|---------|----------|---------|
| 256d | 256 B | 768 GB | 92% | 1.8 TB ✅ |
| 384d (v1.0 选择) | 384 B | 1.15 TB | 95% | 2.0 TB ⚠️ |
| **512d (v1.1 新选择)** | **512 B** | **1.54 TB** | **97%** | **2.2 TB ⚠️** |
| 768d | 768 B | 2.3 TB | 98% | 3.0 TB ❌ |

**v1.1 决策**: **512d SQ8 + PCA Whitening** (精度优先)

**变更理由**:
- v1.0 选 384d 是为严守 2TB RAM, 但 Recall@10 停在 95-97%
- v1.1 目标 98.5%, 必须升级到 512d (+1% Recall)
- 配合 PCA Whitening, 512d 比直接 PCA 多 +0.7%
- RAM 超支 200 GB, 通过缩减 L2 热数据存储补回

**PCA Whitening 说明**:
```python
# 普通 PCA (v1.0)
projected = raw @ pca_matrix

# PCA + Whitening (v1.1, 零成本升级)
# 拟合时额外记录 eigenvalues
projected = (raw @ pca_matrix) / np.sqrt(eigenvalues)
# 效果: Recall@10 +0.5~1%, 零推理开销
```

**备用降级路径** (RAM 不够时):
- 降到 384d + Whitening: 保留 v1.0 部分精度收益, 约 97.5%
- 或热数据用 512d, 冷数据用 256d (混合维度)

### 3.3 索引选择 (v1.1 更新)

| 索引 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| IVF-PQ | 压缩比高 | **亿级误差累积严重** | ❌ |
| HNSW (FP32) | 召回最高 | 存储大 | ❌ |
| **HNSW_SQ8 + refine FP16** | **平衡精度与存储, Milvus 原生** | 略慢于 FP32 | ✅ **主索引** |
| HNSW_PQ | 存储最小 | 精度下降大 | ❌ |
| ScaNN | Google 生产级 | Milvus 支持不成熟 | ❌ |
| **DiskANN** | **SSD 支持十亿级** | RAM 吃 ~40% | ✅ **冷层** |

**v1.0 主索引参数**: `HNSW_SQ8 + refine=FP16, M=32, efConstruction=500`

**v1.1 主索引参数** (升级):
```
HNSW_SQ8:
  M = 48                  # v1.0: 32  (+30% 邻接存储, +0.5% Recall)
  efConstruction = 1000   # v1.0: 500 (+100% 建索引时间, +0.2% Recall)
  ef_search = 300         # v1.0: 200 (+50% 查询延迟, +0.3% Recall)
  sq_type = "SQ8"
  refine = True
  refine_type = "FP16"
  refine_k = 3.0          # v1.0: 2.0 (+50% refine 候选, +0.2% Recall)
```

**参数升级说明**:
- **M=48**: 每个节点存 48 个邻居 (v1.0 是 32)。召回率关键参数。
- **efConstruction=1000**: 建索引时每节点搜索 1000 个候选加邻。一次性成本。
- **ef_search=300**: 查询时遍历深度. 召回率与延迟直接挂钩。
- **refine_k=3.0**: Milvus 内部先用 SQ8 取 Top 300×3=900, 再用 FP16 精排出 Top 300。

**冷层**: `DiskANN, R=48, L_build=100, L_search=80` (不变)

### 3.4 量化策略

| 方案 | 单图 | 精度损失 | 决策 |
|------|------|---------|------|
| FP32 原始 | 1536 B | 0% | ❌ 存储过大 |
| FP16 | 768 B | <1% | ⚠️ 仅用于 L2 热数据 |
| **SQ8 + refine FP16** | **384 B (主) + 768 B (副)** | **~2%** | ✅ **L1 主索引** |
| PQ (24B) | 24 B | 10-15% | ❌ 亿级失真 |

**关键**: Milvus 2.5 的 `HNSW_SQ` 索引支持 `refine` 参数, 查询时内部做"SQ8 剪枝 + FP16 精排", 是官方推荐的两级存储实现。

### 3.5 过滤策略

**场景**: 商品图关联多个商家 ID, 查询时需要 "属于商家 A/B/C 之一" 的过滤。

| 方案 | 实现 | 问题 |
|------|------|------|
| 先 ANN 后过滤 | 检索 Top 1000 → 过滤 | **召回崩塌**, Top 1000 可能无符合商家 |
| 先过滤后搜 | 查询前 bitmap 交集 → 扫描 | 大商家集合时慢 |
| **算子下推 (操作符融合)** | **HNSW 遍历时实时 bitmap.test()** | ✅ **最优** |

**技术栈**: `RoaringBitmap` (由 Daniel Lemire 原版实现, Milvus 2.5 内置支持)

```python
# 过滤实现
merchant_bitmap = RoaringBitmap([img_ids_belonging_to_merchant_123])
search_expr = f"merchant_id in [123, 456, 789]"
# Milvus 自动下推到 HNSW 遍历时的节点过滤
```

**性能**: 单次 bitmap.test() 在 AVX-512 下 1-3 ns, HNSW 遍历 ~1000 节点, 总过滤开销 <5ms。

### 6.4 PCA + Whitening 流程 (v1.1 新增)

```python
import numpy as np
from sklearn.decomposition import PCA

# === 训练阶段 (一次性, 用 10-100 万样本) ===
def fit_pca_whitening(sample_vectors, target_dim=512):
    """
    sample_vectors: shape (N, 1024), DINOv2 原始输出
    """
    # 标准 PCA
    pca = PCA(n_components=target_dim, whiten=True)  # whiten=True 是关键
    pca.fit(sample_vectors)

    # 保存
    np.save("/data/imgsrch/models/pca_512_whiten.npy", {
        "components": pca.components_,           # (512, 1024)
        "mean": pca.mean_,                       # (1024,)
        "explained_variance": pca.explained_variance_,  # (512,)
    })
    return pca

# === 推理阶段 ===
def apply_pca_whitening(raw_vec_1024, pca_data):
    """
    raw_vec_1024: DINOv2 原始输出 (1024,)
    返回: 512d whitened vector
    """
    # 减去均值
    centered = raw_vec_1024 - pca_data["mean"]
    # 投影
    projected = centered @ pca_data["components"].T
    # Whitening: 除以标准差
    whitened = projected / np.sqrt(pca_data["explained_variance"])
    # L2 归一化 (for cosine similarity)
    return whitened / np.linalg.norm(whitened)
```

## 13. Recall vs Precision 概念澄清 (v1.1 新增)

### 13.1 两个指标的本质区别

在图像检索中, 这两个指标衡量不同维度的"质量":

| 指标 | 定义 | 衡量 | 决定者 |
|------|------|------|-------|
| **Recall@K** | 前 K 个结果中**包含**目标的概率 | "能不能找到" | **L1 召回阶段** |
| **Precision@K** | 前 K 个结果中**正确**结果的比例 | "找到的质量" | **L2 精排阶段** |

**数学定义**:

```
假设一次查询的正确答案是集合 T (Ground Truth)
系统返回前 K 个结果 R_K

Recall@K    = |T ∩ R_K| / |T|       # 找到了多少比例的正确答案
Precision@K = |T ∩ R_K| / K         # 返回结果中多少是正确的
```

**关键理解**: 一个系统可以高 Recall 低 Precision (大海捞针捞到了但排在第 80 位), 也可以相反。

### 13.2 实例场景

**场景**: 用户上传一张图, 搜索相似商品。已知数据库里有 3 张该款商品的不同角度照片。

假设返回 Top 10:
- 情况 A: Top 10 里有 2 张正确 (第 3 位和第 7 位)
  - Recall@10 = 2/3 = 66.7%  (3 张里找到 2 张)
  - Precision@10 = 2/10 = 20% (10 个里 2 个对)
- 情况 B: Top 10 里有 3 张正确 (全部前 3 位)
  - Recall@10 = 3/3 = 100%
  - Precision@10 = 3/10 = 30%

### 13.3 对 H3+ 的意义

H3+ 的**最终 Recall@10 ≈ 98.5%** 的含义是:

> "对于 98.5% 的查询, 系统能在 Top 10 里找到至少一个正确答案"

**但这不是说 Top 10 全对**。实际上:
- Recall@10 ≈ 98.5% (至少一个对)
- Recall@1 ≈ 90% (第一个就对)
- Precision@10 ≈ 70-80% (Top 10 里 7-8 个是高度相似)

### 13.4 H3+ 各层指标分解

```
┌─────────────────────────────────────────────────┐
│  L1 HNSW_SQ8 召回 (v1.1 ef_search=300, Top 300) │
│                                                  │
│  Recall@300:    99.5%  ← 拉进候选池的上限        │
│  Recall@100:    98.5%                            │
│  Recall@10:     95%    (未精排)                  │
│  Recall@1:      70%    (未精排)                  │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│  L2 FP16 精排 Top 300 → Top 10                  │
│                                                  │
│  Recall@10:     98.5%  ← 最终目标                │
│  Recall@1:      90%                              │
│  Precision@10:  70-80%                           │
└─────────────────────────────────────────────────┘
```

**关键观察**:

1. **L1 Recall@300 是系统上限 (99.5%)** — 如果目标没进 L1 的 Top 300, L2 再好的精排也救不回来。
2. **L2 精排把 Recall@10 从 95% 拉到 98.5%** — 精排真正的价值。
3. **Precision@10 是独立指标** — 用户体验的 "Top 10 质量", 当前约 70-80%, 提升需要更好的语义模型或 re-rank。

### 13.5 常见误解

| 误解 | 正确理解 |
|------|---------|
| "Recall@10 = 98.5% 意味着 Top 10 有 9.85 个对的" | 错误。是"至少 1 个对的概率" |
| "提高 ef_search 能提高 Precision" | 部分正确。主要提高 Recall, 对 Precision 间接影响 |
| "精排就是提高召回率" | 错误。精排是把已经在候选池里的正确答案**排到前面**, 不增加新候选 |
| "两个都是越高越好" | 对, 但优化路径不同。Recall 靠 L1, Precision 靠 L2 |

## 14. 召回率专项优化路径 (v1.1 新增, 超越 98.5% 的路线图)

这一章记录 **v1.1 之后**, 如果业务需要进一步提升召回率的升级方案。

### 14.1 提升等级对比

| 目标 | 当前 | 成本 | 用户感知 | 推荐度 |
|------|------|------|---------|-------|
| **97% → 98.5%** | **v1.1 (H3+)** | **低** | **中等** | ✅ **已采纳** |
| 98.5% → 99% | v1.2 候选 | 中 | 小 | ⏭️ 未来 |
| 99% → 99.5% | v1.3 候选 | 高 | 极小 | ❌ 不推荐 |
| 99.5% → 99.9% | v1.4 候选 | 极高 | 几乎无 | ❌ 不推荐 |

### 14.2 v1.2 候选方案 (98.5% → 99%, 中成本)

**仅在 v1.1 实测后发现 98.5% 不够时启用**。

#### 方案 14.2.1: 扩展候选数 (最简单)

```
改动:
  ef_search:       300 → 500
  精排候选数:      300 → 500
  refine_k:        3.0 → 4.0
  
效果:  +0.2-0.3% Recall@10
成本:  延迟 +8ms (36ms → 44ms)
存储:  无变化
```

#### 方案 14.2.2: 多路召回 (针对热数据)

```
改动:
  热数据 (2亿) 额外存 CLIP 向量 (256d SQ8)
  查询时 DINOv2 + CLIP 并行召回, 合并去重
  加权: 0.7 * DINOv2 + 0.3 * CLIP
  
效果:  +0.8% Recall@10 (热数据), +0.2% 平均
成本:  热数据存储 +64 GB
延迟:  并行召回 +3ms
训练:  无
```

#### 方案 14.2.3: 多尺度查询端增强

```
改动:
  对查询图生成 4 个视图:
    - 原图 224×224
    - 中心裁剪 80%
    - YOLO 主体裁剪 (DeepFashion2)
    - 上半/下半图 (针对服装)
  每个视图单独搜索, 合并去重
  
效果:  +0.5% 通用, +2-3% 截图场景
成本:  延迟 +10ms (并行) 或 +40ms (串行)
存储:  无变化
```

#### v1.2 组合方案

```
v1.1 基础 (98.5%) +
  ✅ 方案 14.2.1 扩展候选数:      +0.3%
  ✅ 方案 14.2.2 多路召回 (热):   +0.2% (平均)
  ⏭️ 方案 14.2.3 多尺度 (可选):   +0.5%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Target: 99-99.5% Recall@10
Cost:   延迟 +15ms, 存储 +64 GB
```

### 14.3 v1.3 候选方案 (99% → 99.5%, 高成本)

**一般不推荐**, 仅在关键业务 (商品防重、版权检测) 必需时启用。

#### 方案 14.3.1: LoRA 微调 DINOv2

```
训练数据: 50万正样本对 (同款商品不同角度)
训练时间: 4-8 小时 (单 GPU)
效果:    +1-2% Recall@10
存储:    无变化 (LoRA 权重 ~10MB)
```

#### 方案 14.3.2: 引入 SSCD 双向量

```
存储: DINOv2 512d + SSCD 256d
适用场景: 截图/副本检测强化
效果:    特定场景 +2-3%, 平均 +1%
存储代价: x1.5 (多存 256d SQ8)
```

#### 方案 14.3.3: Query Expansion (查询扩展)

```
第一轮搜索 → Top 5 结果
扩展向量 = 0.7 * query + 0.3 * mean(Top 5)
第二轮搜索 → 最终结果

效果:  +0.5% Recall@10
延迟:  x2 (两次搜索)
风险:  Top 5 有错误时放大错误
```

### 14.4 v1.4 候选方案 (99.5%+, 极高成本)

#### 完整微调 (Metric Learning)

```
数据:    100万 triplet (anchor, positive, negative)
训练:    2-3 天
方法:    ArcFace / Triplet Loss
效果:    +2-5% Recall@10
风险:    过拟合, 泛化性下降
```

#### Cross-Encoder 终极重排

```
对 Top 10 用大模型 (BLIP-2 / SigLIP-2) 两两比较
效果:    +0.3-0.5% Recall@1 (精排质量而非召回)
延迟:    +100-200ms
```

### 14.5 升级决策流程

```
v1.1 上线 → Phase 2 实测 Recall@10 ≈ ?
  │
  ├─ ≥ 98.5% (达标) ────────────────────→ 维持 v1.1
  │
  ├─ 97-98.5% (接近)                    
  │   │
  │   ├─ 业务可接受 ─────────────────→ 维持 v1.1
  │   └─ 业务不接受 ─────────────────→ 启用 v1.2 (扩候选数)
  │
  └─ < 97% (未达标)
      │
      ├─ 检查: 是否符合实施细节?
      │   └─ 否 → 修正实施
      │
      └─ 是 → 启用 v1.2 完整版 (多路召回)
              │
              └─ 若仍不够 → 启用 v1.3 (LoRA 微调)
```

### 14.6 成本收益对比表 (完整路线图)

| 版本 | Recall@10 | 存储/图 | 延迟 (热) | 增量成本 | 采纳建议 |
|------|----------|--------|----------|---------|---------|
| v1.0 基础 | 97% | 1380 B | 26ms | - | ❌ 已升级 |
| **v1.1 (本文档)** | **98.5%** | **1851 B** | **36ms** | **低** | ✅ **采纳** |
| v1.2 扩候选 | 98.8% | 1851 B | 44ms | 低 | ⏭️ 可选 |
| v1.2 完整 (多路) | 99.3% | 1915 B | 50ms | 中 | ⏭️ 可选 |
| v1.3 (LoRA) | 99.5% | 1851 B | 36ms | 中 (训练) | ❌ 暂不 |
| v1.4 (全微调) | 99.8% | 1851 B | 36ms | 高 (训练) | ❌ 暂不 |

**决策原则**: 每次升级只解决一个瓶颈, 用真实数据指导。**不要盲目追求 99%+**。
