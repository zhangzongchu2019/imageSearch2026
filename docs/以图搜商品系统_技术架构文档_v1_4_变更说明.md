# 以图搜商品系统 — 技术架构文档 v1.4 变更说明

> 版本: v1.4
> 日期: 2026-03-31
> 基线: v1.3 (2026-02-10)
> 关联: BRD v5.0 / 搜索需求文档 v2.0 / 过滤召回架构分析 v2.0

---

## 变更概述

v1.4 是架构级重构，核心变更源于两个关键发现：

1. **图片-商家多对多关系 (转发倍率 24x)**: merchant_id 不能作为 Milvus 标量预过滤字段，bitmap 基础设施必须保留
2. **商家级搜索选择率极低 (0.00025%)**: 全局 ANN + bitmap 后过滤对小商家搜索召回率趋近于零，需要全新的搜索策略

## 二、核心场景定义 (新增)

v1.3 未明确区分搜索场景，v1.4 定义三个核心场景:

| 场景 | 搜索范围 | 图片数 | 响应方式 | 速度 | 精度 | 流量 |
|------|---------|--------|---------|------|------|------|
| **1. 图搜开单** | 本商家 (~1万图) | 1 张 | 一次性返回 | **≤ 2s 端到端** | Top-1 必须准 | ~50% |
| **2. 拿图找货** | 关注上家 (10~3000家) | 1 张 | 一次性返回 | 可放宽 | 找得到+准 | ~50% |
| **3. 批量查询** | 同 1 或 2 | ≤ 7 张 | **SSE 流式** | 首张尽快 | 准确 | 含在 1/2 中 |

## 三、分区策略变更 (原 §4.4 重写)

### 3.1 v1.3 → v1.4 分区策略对比

| | v1.3 | **v1.4** |
|---|---|---|
| 分区维度 | 两区 (热区5月 + 非热区13月 + 常青) | **标准月分区 19 + SVIP 月分区 19 = 38 个** |
| 热区 | 近 5 个月 HNSW, 单一集合 | **近 6 个月 HNSW, 按月独立分区** |
| 冷区 | 6-18 月 DiskANN, 单一集合 | **7-18 月 DiskANN, 按月独立分区** |
| 常青 | 非热区内独立分区 | **独立月分区 (标准+SVIP 各 1 个)** |
| SVIP | 不存在 | **≥1000 万图的头部商家独立月分区** |
| TTL 清理 | Milvus Delete + compaction | **drop 月分区, 干净利落** |

### 3.2 新分区布局

```
Milvus Collection: global_images

标准月分区 (普通商家, < 1000 万图):
├── month_202601  (热区, HNSW)
├── month_202602  (热区, HNSW)
├── ...
├── month_202606  (热区, HNSW)     ← 热区 6 个月
├── month_202507  (冷区, DiskANN)
├── ...
├── month_202412  (冷区, DiskANN)  ← 冷区 12 个月
├── month_evergreen                 ← 常青
│                                   小计: 19 个分区

SVIP 月分区 (≥ 1000 万图, ~16 个 SVIP 共享):
├── svip_202601   (热区, HNSW)
├── ...
├── svip_202606   (热区, HNSW)
├── svip_202507   (冷区, DiskANN)
├── ...
├── svip_202412   (冷区, DiskANN)
├── svip_evergreen
│                                   小计: 19 个分区

合计: 38 个分区
```

### 3.3 SVIP 路由表 (新增 PG 表)

```sql
CREATE TABLE svip_partition_roster (
    merchant_id   VARCHAR(64)  NOT NULL,
    ts_month      INT          NOT NULL,   -- 202601
    partition_type VARCHAR(10)  NOT NULL,   -- 'svip' | 'standard'
    svip_pool_id  INT,
    created_at    TIMESTAMP    DEFAULT now(),
    PRIMARY KEY (merchant_id, ts_month)
);
CREATE INDEX idx_roster_month_type ON svip_partition_roster(ts_month, partition_type);
```

- 路由查询缓存 Redis: `route:{merchant_id}:{ts_month}`, TTL 1h
- 写入/搜索均通过路由表判断分区归属
- 晋升/降级从下月生效，不迁移历史数据

### 3.4 分区管理规则 (替代 v1.3 §4.4)

| 操作 | 时机 | 标准分区 | SVIP 分区 |
|------|------|---------|----------|
| 创建新月分区 | 每月 1 日 | month_YYYYMM | svip_YYYYMM |
| 冷热切换 | 第 6→7 月 | HNSW → DiskANN 索引重建 | 同 |
| TTL 清理 | 每月 1 日 | **drop 过期月分区** | **drop 过期月分区** |
| 索引构建 | 新分区创建后 | build index (HNSW/DiskANN) | 同 |
| 内存管理 | 热区 load, 冷区 release | 6 个月常驻 | 同 |

## 四、检索引擎变更 (原 §5.1 重写)

### 4.1 核心变更: 自适应搜索策略

v1.3 所有商家走统一的 "ANN → bitmap 后过滤" 路径。v1.4 根据商家规模自适应:

```
                    搜索范围判定
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
     小范围 (≤100万pk)  中范围       大范围 (>1000万pk)
     含场景1单商家     (100~1000万)   场景2大范围
            │            │            │
            ▼            ▼            ▼
     bitmap 预查      Milvus ANN    Milvus ANN
     image_pk 列表    + bitmap      + bitmap
     → Milvus 取向量  后过滤         后过滤
     → 暴力搜索                     (SVIP 分区内)
            │            │            │
            ▼            ▼            ▼
        < 10ms        50~150ms     100~300ms
        精度100%       精度~98%     精度~95%
```

### 4.2 新检索流水线

```
Stage 0   : 降级覆盖 + merchant_scope 解析 + 商家规模判定
Stage 0.5 : bitmap 预构建 / pk 列表预查 (与 Stage 1 并行)
Stage 1   : 特征提取 (GPU FP16)
            ↓
            ┌── 小范围路径 ──────────────────────── 大范围路径 ──┐
            │                                                    │
Stage 2   : Milvus 按 pk 取向量           热区 ANN (HNSW)       │
            + 进程内暴力搜索               + 标签召回 (并行)      │
            ↓                              ↓                     │
Stage 2-NH: (不需要级联)                   [级联] 非热区 DiskANN  │
            ↓                              ↓                     │
Stage 3   : (不需要 bitmap 过滤)           bitmap 后过滤          │
            └─────────────┬────────────────┘                     │
                          ↓
Stage 4   : 全局 + 子图 **双路并行搜索** + Refine 精排
Stage 5   : 匹配层级分类 (P0/P1/P2) + SPU 聚合
Stage 6   : 响应构建 + 异步日志
```

### 4.3 子图搜索: 从 fallback 升级为必备并行路径

| | v1.3 | **v1.4** |
|---|---|---|
| 定位 | 条件触发的 fallback (score < 0.75) | **必备双路并行, 每次搜索都执行** |
| 执行时机 | ANN 后, 仅极少数请求 | **与全局 ANN 同时并行** |
| 查询侧子图提取 | Lazy 模式, 仅 fallback 触发 | **写入时已全量提取, 搜索时直接用** |
| 原因 | — | 截取/子目标/局部翻拍是刚需场景, 非边缘情况 |

### 4.4 匹配层级 (新增)

| 优先级 | 匹配类型 | 相似度 | confidence | 业务含义 |
|--------|---------|--------|-----------|---------|
| P0 | 原图匹配 | > 0.90 | HIGH | 可直接开单 |
| P1 | 款式匹配 (同款不同色) | 0.70~0.90 | MEDIUM | 需确认后开单 |
| P2 | 品类+风格一致 | 0.50~0.70 | LOW | 仅供参考 |

结果按 P0 > P1 > P2 排序，同层级内按相似度降序。

### 4.5 SPU 聚合 (新增)

按 product_id 聚合: 同一商品的多张图命中任一张，归到同一商品展示，不重复。每个 SPU 取最高匹配分的图片作为代表。

### 4.6 标签搜索 (新增独立入口)

```
POST /api/v1/tag/search
  输入: tags[], merchant_scope, top_k
  逻辑: Milvus INVERTED 索引查询, 按标签命中数降序排列
  同样受 merchant_scope 范围约束
  同样使用 bitmap 预查 / 后过滤的自适应策略
```

### 4.7 批量搜索 + SSE 流式返回 (新增)

```
POST /api/v1/image/search/batch
  输入: query_images[] (≤7 张), merchant_scope, top_k
  输出: SSE (Server-Sent Events) 流式返回
  逻辑:
    1. 7 张图 batch 特征提取 (1 次 GPU 调用)
    2. 7 次搜索并行执行, 共享 merchant_scope 解析
    3. 任一搜索完成即刻推送结果
    4. 首张结果延迟 ≤ 500ms~1.5s
```

## 五、商家过滤子系统变更 (原 §5.4 修订)

### 5.1 bitmap 角色升级

| | v1.3 | **v1.4** |
|---|---|---|
| 主要角色 | ANN 后唯一过滤手段 | **双角色: 前置预查 + 后置过滤** |
| 新增能力 | — | **按商家查 pk 列表 (反向索引)** |
| 基础设施 | 全保留 | **全保留, RocksDB 新增反向索引** |

### 5.2 bitmap 新增前置预查能力

```
现有: filter(candidate_pks, merchant_scope) → filtered_pks   (后过滤)
新增: get_image_pks(merchant_id) → [pk1, pk2, ..., pkN]       (前置预查)
```

RocksDB 新增 Column Family:

```
cf_merchant_index:
  Key:   merchant_bitmap_index (uint32, 4 bytes)
  Value: image_pk 列表 (序列化, 变长)
  同步: 与 cf_rolling / cf_evergreen 同步更新
```

### 5.3 过滤降级层级 (修订)

| 层级 | v1.3 | **v1.4** |
|------|------|---------|
| Level 0 | bitmap-filter gRPC (P99 ≤ 8ms) | 小商家: bitmap 预查 pk + Milvus 取向量 + 暴力搜索 |
| Level 0' | — | **大范围: Milvus ANN + bitmap-filter gRPC 后过滤** |
| Level 1 | 远端 PG 查询 (P99 ≈ 15ms) | 同 v1.3 |
| Level 2 | 强制 S2/S4 降级 | 同 v1.3 |

## 六、降级策略变更 (原 §5.5 修订)

### 6.1 S1 状态变更

| | v1.3 S1 | **v1.4 S1** |
|---|---|---|
| enable_fallback | false | **true (保留多路召回)** |
| enable_cascade | false | false (同) |
| ef_search | 128 | **192** |
| refine_top_k | 1000 | **2000** |

### 6.2 降级阈值变更

| 状态 | v1.3 触发条件 | **v1.4 触发条件** |
|------|-------------|-----------------|
| S0→S1 | P99>450ms 持续 2min | **P99>600ms 持续 3min** |
| S0→S2 | P99>800ms 持续 30s | **P99>1000ms 持续 1min** |

原则: **召回率优先, 容忍更高延迟, 更慎重进入降级。** S1 保留 fallback 保证多路召回能力, 仅 S2 才全面裁剪。

## 七、写入引擎变更 (原 §5.2 补充)

### 7.1 写入可见性

v1.3 定义组件级 SLA (5s/10s/15s), v1.4 补充端到端 SLA:

```
端到端 ≤ 3 分钟 = 上传 (27~90s) + 合规审查 (45s) + 入库 (≤45s)
单张入库 ≤ 5s (组件级, 与 v1.3 Level 1 一致)
```

### 7.2 写入路由

写入时通过 SVIP 路由表判断目标分区:

```python
partition = await get_write_partition(merchant_id, ts_month)
# 返回 "month_202601" 或 "svip_202601"
```

## 十二、云端灾备最小功能集 (v1.4 新增)

### 12.1 定位

云端独立部署的灾备实例，保留最近 **1 个月**数据，在 IDC 主站故障时保障**图搜开单核心场景**不中断。

### 12.2 数据规模

| 指标 | 值 |
|------|-----|
| 数据范围 | 最近 1 个月 |
| 图片量 | **~1.4 亿张** (月增量全量) |
| 全局向量 (256d × 4B) | ~134 GB |
| 子图向量 (128d × 4B × 3) | ~200 GB |
| 标量 + HNSW 索引 | ~70 GB |
| **Milvus 总计** | **~400 GB** |
| PG bitmap | ~30 GB |
| **存储总计** | **~430 GB** |

### 12.3 架构

```
IDC 主站 (全功能)                      云端灾备 (最小功能集)
┌─────────────────────┐               ┌─────────────────────┐
│ 38 分区, 30亿图       │   实时同步     │ 2 分区 (month+svip)  │
│ Milvus 集群          │ ───Kafka───→  │ Milvus 单节点 512GB  │
│ PG + RocksDB         │               │ PG 精简              │
│ 全功能搜索            │               │ 仅暴力搜索            │
│ 100 QPS              │               │ ~30 QPS              │
└─────────────────────┘               └─────────────────────┘
```

### 12.4 实时同步

```
IDC write-service
  ↓ (每笔写入)
Kafka (image_events + merchant_events)
  ↓ (跨机房 MirrorMaker)
云端 Kafka
  ↓
云端 sync-consumer
  ├── Milvus 写入 (仅当月分区)
  ├── PG bitmap 写入
  └── 每月 1 日: 创建新月分区, drop 上上月分区
```

| 同步指标 | 目标 |
|---------|------|
| 同步延迟 | P95 ≤ 30 秒 |
| 数据完整性 | 以 IDC Kafka 为准, at-least-once |
| 月度轮转 | 云端自动创建/drop, 始终保留最近 1 个月 |

### 12.5 功能范围

| 功能 | 支持 | 说明 |
|------|------|------|
| 场景 1 图搜开单 | **是** | 核心保障，暴力搜索精度 100% |
| 场景 2 拿图找货 | **有限** | 仅最近 1 月数据，覆盖不全 |
| 场景 3 批量查询 | **是** | 复用同一逻辑 |
| 标签搜索 | **是** | |
| 级联非热区 | **否** | 无冷区数据 |
| 常青池 | **否** | 不同步 |
| 子图并行 | **可选** | 按云端资源决定 |
| 匹配层级 P0/P1/P2 | **是** | |
| SPU 聚合 | **是** | |

### 12.6 灾备切换

```
正常: DNS → IDC 主站

IDC 故障 (健康检查连续 3 次失败, ~15 秒):
  → DNS failover → 云端灾备
  → 用户自动切换, 功能降级但服务不中断
  → 搜索范围缩小到最近 1 个月 (meta.degraded=true)

IDC 恢复:
  → 人工确认后 DNS 切回 IDC
  → 云端期间增量数据通过 Kafka offset 回补 IDC
```

### 12.7 降级状态: S_MIN

| 状态 | 触发 | 数据范围 | 搜索策略 |

| S2 | P99>1000ms | 热区 + 常青 | 禁用级联 + fallback |
| **S_MIN** | **IDC 故障, 云端接管** | **最近 1 个月** | **仅暴力搜索** |
