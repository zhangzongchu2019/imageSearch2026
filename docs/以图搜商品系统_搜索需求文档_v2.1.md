## 变更详情

### 1. 新增: 品类加权搜索策略

**v2.0 状态**: 融合排序公式为固定权重 (global 0.40 + sub 0.30 + tag 0.20 + cat 0.10)，品类匹配为布尔值。

**v2.1 新增**:

品类匹配从布尔值细化为分级加权:

| 品类匹配层级 | 加权分值 | 说明 |
|-------------|---------|------|
| 同 L1 品类 | +0.05 | 查询图与结果图属于相同一级品类 (如都是"箱包类") |
| 同 L2 品类 | +0.03 | 在 L1 匹配基础上，二级品类也一致 (如都是"单肩包") |

加权作用于最终排序分数:

```
final_score = raw_similarity_score + l1_bonus + l2_bonus

示例:
  raw_score = 0.85, 同 L1 + 同 L2 → final = 0.85 + 0.05 + 0.03 = 0.93
  raw_score = 0.85, 仅同 L1       → final = 0.85 + 0.05 = 0.90
  raw_score = 0.85, 品类不同       → final = 0.85
```

### 2. 新增: 两路搜索合并策略

**v2.0 状态**: 检索管线描述了多阶段流程，未明确品类过滤路径的合并方式。

**v2.1 新增**:

搜索执行两路并行，结果合并去重:

| 搜索路径 | 策略 | 返回数量 |
|---------|------|---------|
| **全局搜索** | 不限品类，全局 ANN 搜索 | Top-20 |
| **品类过滤搜索** | 限定与查询图同 L1 品类，ANN 搜索 | Top-20 |

合并规则:
1. 两路结果合并 (最多 40 条)
2. 按 image_id 去重 (保留分数较高者)
3. 应用品类加权 (+0.05/+0.03)
4. 按 final_score 降序排序
5. 应用相似度阈值过滤 (≥0.80)
6. 返回最终 Top-K

两路合并的目的: 全局搜索保证召回多样性，品类过滤搜索保证同品类结果的深度覆盖。

---

### 3. 新增: 相似度阈值

**v2.0 状态**: P0/P1/P2 匹配层级定义了典型相似度范围，但无硬截断。

**v2.1 新增**:

| 项目 | 说明 |
|------|------|
| 阈值条件 | raw_score >= 0.80 |
| 过滤时机 | 品类加权之前，基于原始相似度分数判断 |
| 过滤行为 | raw_score < 0.80 的结果不返回给前端 |
| 业务含义 | 低于 0.80 的结果通常为不同品类或不同风格，展示会降低用户信任 |

与匹配层级的关系:

| 匹配层级 | 原定义 | 阈值后 |
|---------|--------|--------|
| P0 原图匹配 | > 0.90 | 保留 (> 0.90) |
| P1 款式匹配 | 0.70~0.90 | 仅保留 0.80~0.90 段 |
| P2 品类风格 | 0.50~0.70 | **全部过滤** |

> 注: P2 结果在 v2.1 中被阈值过滤，不再展示。后续可根据业务反馈调整阈值。

### 5. 新增: 搜索精度基线

**v2.0 状态**: 未建立搜索精度基线数据。

**v2.1 新增**:

基于全链路测试 (Top-8, 查询图: 短袖T恤) 建立的精度基线:

| 指标 | 结果 | 说明 |
|------|------|------|
| L1 品类准确率 | **100%** | Top-8 结果全部为"服装类" |
| Top-5 精准率 | **40%** | Top-5 中 2 条为短袖T恤 (精确匹配) |
| Top-8 品类一致率 | 100% | 所有返回结果品类正确 |

> 基线说明: 当前精准率受限于库内数据多样性和模型细粒度区分能力。品类级别准确率优秀，细粒度 (同款/同型号) 精准率有提升空间。

## 二、业务特征

1. **商家品类高度集中** — 服装鞋帽箱包商家按季节关注 2~5 个品类，珠宝玉器商家几乎单品类
2. **商家间为上下游关系** — 关注的商家都是同行业，搜索空间天然内聚
3. **图片分布极度长尾** — 典型商家 1 万张 vs 头部商家 800 万张，差距 1000 倍

## 三、搜索方式

| 搜索方式 | 输入 | 检索方式 | 状态 |
|---------|------|---------|------|
| **以图搜图** | 图片 | 向量 ANN 检索 | 核心功能 |
| **以标签搜图** | 多标签 | 标签交集，命中越多排越前 | 核心功能 |
| 图 + 标签混合 | 图片 + 标签 | 向量 + 标签联合 | 预留，暂不实现 |

## 四、使用场景

### 场景 1：图搜开单

- **业务目的**: 客户发图，找到商品，开单发货
- **搜索范围**: 本商家 (~1 万张)
- **图片数**: 1 张
- **响应方式**: 一次性返回
- **速度要求**: ≤ 2 秒端到端
- **精度要求**: 必须找对（发错货是业务事故）
- **流量占比**: ~50%

### 场景 2：拿图找货

- **业务目的**: 在上家中找货源
- **搜索范围**: 自己 + 关注上家 (10~3000+ 家)
- **图片数**: 1 张
- **响应方式**: 一次性返回
- **速度要求**: 可放宽
- **精度要求**: 找得到 + 够精准
- **流量占比**: ~50%

### 场景 3：批量查询

- **业务目的**: 批量处理多张图
- **搜索范围**: 同场景 1 或 2
- **图片数**: ≤ 7 张
- **响应方式**: 流式逐张返回（先完成先推送）
- **速度要求**: 首张结果尽快返回
- **精度要求**: 准确
- **流量占比**: 覆盖在场景 1、2 中

## 五、Query 图的变形情况

| 变形类型 | 说明 | 要求 |
|---------|------|------|
| 原图缩放/有损压缩 | 图片经过转发、压缩 | 必须命中 |
| 截取局部 | 只包含商品某个部分 | 必须命中 |
| 原图拉伸 | 比例变形 | 必须命中 |
| 翻拍（不同角度） | 对着屏幕拍照 | 必须命中 |
| 原图中的子目标 | 如模特穿了上衣+裤子，query 只有裤子 | 必须命中 |
| 局部翻拍+遮挡 | 对原图拍照，只拍了一部分，且有反光/阴影等遮盖 | 人眼可识别范围内必须召回 |

**核心要求: 全局向量 + 子图向量双路搜索为必备能力，非可选 fallback。**

**鲁棒性基准: 人眼可识别 = 系统应可召回。** 局部翻拍+遮挡是最严苛的变形场景，同时叠加了截取、视角变化、光照干扰三重因素，对特征提取模型的鲁棒性要求最高。

## 六、结果匹配层级与排序

| 优先级 | 匹配类型 | 典型相似度 | 业务含义 |
|--------|---------|----------|---------|
| **P0** | 原图匹配 | >0.90 | 可直接开单 |
| **P1** | 款式匹配（同款不同色） | 0.70~0.90 | 需确认后开单 |
| **P2** | 品类 + 风格一致 | 0.50~0.70 | 仅供参考选品 |

- 结果应按层级分层呈现，P0 永远排最前
- confidence 标识需与业务层级对齐: HIGH=可开单 / MEDIUM=需确认 / LOW=参考

## 七、标签搜索要求

- 支持多标签输入
- 按标签命中数量排序，命中越多越靠前
- 同样受 merchant_id 范围约束（场景 1/2/3 均适用）

## 八、写入链路要求

每张商品图入库时必须完成:

```
原图 → 全局向量提取
    → 目标检测 (上衣/裤子/包/鞋等) → 各子图向量提取
    → 标签提取
    → 全部入库 (原图向量 + 子图向量 + 标签)
```

**写入可见性要求: 用户上新后，≤ 3 分钟端到端可被搜索到。**

单次上新最多 9 张图，3 分钟时间预算拆解:

| 环节 | 耗时 | 说明 |
|------|------|------|
| 手机上传 | 3~10 秒/张 × 9 张 = 27~90 秒 | 客户端网络 |
| 合规审查 | 5 秒/张 × 9 张 = 45 秒 | 内容安全 |
| **入库写入** | **≤ 45 秒 (9张合计)** | 向量提取+子图检测+标签+Milvus写入+索引生效 |

入库写入预算 45 秒 / 9 张 ≈ **每张 ≤ 5 秒**，需完成: 全局向量提取 → 子图检测+向量提取 → 标签提取 → Milvus 入库 → 可被查询命中。

---

## 九、架构关键约束

| 约束 | 原因 |
|------|------|
| `merchant_id` 标量预过滤 | 全局 ANN + 后过滤召回率 ≈ 0%，必须前置缩小范围 |
| 全局 + 子图双路并行搜索 | 截取/子目标场景为刚需 |
| 场景 3 流式返回 (SSE) | 首张结果尽快反馈 |
| 按 SPU 聚合 | 同一单品多张图命中任一张应归到同一商品 |

---

## 十、搜索原则

1. **保证召回率和准确性为主，调用执行时间次之**
2. **能在调用前缩小范围就先缩小范围**
