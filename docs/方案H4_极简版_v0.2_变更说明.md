- **基线**: H4 v0.1
- **评审来源**: ChatGPT 对 v0.1 的 5 点评审
- **修订日期**: 2026-04-10

## 变更 1: merchant_ids 从主表拆出

### v0.1 (修改前)

```python
# 主表字段
FieldSchema(name="merchant_ids", dtype=DataType.ARRAY, 
            element_type=DataType.INT32, max_capacity=100)
# 索引
merchant_index = {"index_type": "BITMAP"}
```

### v0.2 (修改后)

```
主表: 只存 merchant_set_id (INT64, 哈希值)

外挂层 (独立 Collection 或外部 Redis/RoaringBitmap):
  merchant_id → RoaringBitmap(image_pk 集合)
  merchant_set_id → Bitmap(merchant_id 集合)

查询流程:
  1. 用户查 merchant_id=123
  2. 从外挂层取 bitmap → 得到 image_pk 集合
  3. 传给 Milvus 做 expr="image_pk in [...]" 或 bitset 下推
```

### 理由

Milvus 官方明确: BITMAP 最适合 distinct values < 500。
商家 ID 可能有几万个, 属于高基数字段, BITMAP 会导致存储膨胀和性能退化。

### 存储影响

```
主表: merchant_ids ARRAY ~30B → merchant_set_id INT64 8B (省 22B/图)
外挂: merchant_id → bitmap, 存 Redis 或独立 Collection
```

## 变更 2: 标签字段分层 (硬过滤 / 软打分)

### v0.1 (修改前)

```python
# 所有 9 个标签字段都参与硬过滤
expr = "category_l2 == T恤 AND color == 红 AND material == 棉 AND style == 休闲"
```

### v0.2 (修改后)

```
强过滤字段 (准确率 >90%, 用 WHERE):
  category_l1    ~95%   "服装" vs "家具" → 很少搞错
  category_l2    ~85%   "T恤" vs "衬衫" → 偶尔混淆但大体对
  color_code     ~90%   "红" vs "蓝" → 视觉明显
  brand_id       100%   业务方提供, 不是 CLIP 推

弱过滤字段 (准确率 <85%, 仅参与 score boost):
  material_code  ~70%   棉 vs 涤纶 → 视觉难分, 不做硬过滤
  pattern_code   ~80%   纯色 vs 条纹 → 偶尔错, 可 boost
  style_code     ~75%   休闲 vs 商务 → 主观, 不做硬过滤
  sleeve_code    ~80%   短袖 vs 长袖 → 大体对, 可 boost
  season_code    ~60%   春夏 vs 秋冬 → 很难判, 只 boost
  gender_code    ~80%   男 vs 女 → 大体对, 可 boost
```

### 搜索公式 (v0.2)

```python
# 硬过滤 (只用强字段)
expr = "category_l1 == 1001 AND color_code == 1"

# 软打分 (弱字段参与加分, 不参与过滤)
boost_score = 0
if result.material == query.material: boost_score += 0.1
if result.pattern == query.pattern: boost_score += 0.1
if result.style == query.style: boost_score += 0.05
...

final_score = 0.8 * sscd_score + 0.2 * boost_score
```

### 理由

CLIP 零样本在细粒度属性 (材质/风格/季节) 准确率只有 60-80%。
如果做硬过滤, 正确结果会被错误标签直接排除, 召回率受损 15-30%。

## 变更 3: 图中图降级为非默认承诺

### v0.1 (修改前)

```
图中图解决方案:
  实时: 中心裁剪 60% → 预期 30-50%
  异步: YOLO → 预期 60-80%
```

### v0.2 (修改后)

```
核心能力 (默认承诺):
  ✅ 原图/压缩/裁剪/形变/涂鸦/缩放: R@10 99%+
  ✅ 粗语义过滤后实例排序
  
增强能力 (非默认, 需额外配置):
  L2: 中心裁剪 → 图中图 30-50% (辅助, 不保证)
  L3: 异步 YOLO 子图 → 图中图 60-80% (可选扩展)

文档明确标注:
  "H4 v0.2 的主目标是实例近重复检索, 不是稳定图中图检索。
   图中图需要 L2/L3 增强, 用户需知悉此限制。"
```

## 变更 4: 存储预算改为区间

### v0.1 (修改前)

```
单图总计: ~1,590 B (~1.6 KB)
```

### v0.2 (修改后)

```
单图存储:
  理论目标:     1.6 KB
  工程预算区间: 1.6 KB ~ 2.1 KB
  
  差异来源:
    - HNSW_SQ 图结构在不同数据分布下开销不同
    - BITMAP/STL_SORT 索引摊销受数据基数影响
    - Milvus segment/partition/meta 固定开销
    
  上线前必须: 用 1000 万级真实数据回归实测实际值

30 亿 / 8 机:
  目标:   596 GB/机 (30%)
  预算:   596 ~ 787 GB/机 (30-39%)
  上限:   不超过 1 TB/机 (50%)
```

## 变更 5: 搜索模式重命名

### v0.1 (修改前)

```
mode=instance   实例搜索
mode=semantic    语义搜索
mode=exact       精确搜索
mode=auto        智能融合
```

### v0.2 (修改后)

```
mode=instance           实例搜索 (SSCD 向量)
mode=filtered_instance  过滤后实例排序 (标签过滤 + SSCD)  ← 原 semantic
mode=exact              精确搜索 (pHash + SSCD)
mode=auto               智能融合 (默认)

说明:
  "filtered_instance" 更准确地描述了能力边界:
  它不是 "用向量做语义相似", 而是 "用标签缩小范围后做实例排序"
  
  真正的语义搜索 (跨款/跨品牌/跨品类) 需要:
  - 存储语义向量 (CLIP/DINOv2)
  - 或更精确的标签系统
  这在 v0.2 中不作为核心承诺
```

## v0.2 能力定位 (明确边界)

### 核心能力 (承诺)

- ✅ 原图 / 压缩 / 裁剪 / 轻形变 / 涂鸦 / 水印 → R@10 99%+
- ✅ 重度裁剪 30% → R@10 99%+ (SSCD 强项)
- ✅ 粗语义过滤后实例排序 (品类+颜色+SSCD)
- ✅ 精确去重 (pHash)

### 增强能力 (非默认)

- ⚠️ 图中图 → 中心裁剪辅助 (30-50%), 需 L2/L3 增强
- ⚠️ 反图中图 → SSCD 57%, 需子图向量增强

### 非目标 (不承诺)

- ❌ 强语义跨款搜索 (需语义向量)
- ❌ 高精度商品属性真值 (CLIP 零样本有限)
- ❌ 稳定子图/图中图搜索 (需 YOLO pipeline)
