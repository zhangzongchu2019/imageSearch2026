**v5.0 状态**: 搜索接口仅支持 base64 图片输入。

| 项目 | 说明 |
|------|------|
| 上传方式 | 腾讯云 COS JavaScript SDK 浏览器端直传 |
| 鉴权方式 | STS (Security Token Service) 临时密钥，有效期 30 分钟 |

| 优势 | 图片不经过应用服务器，减轻带宽压力；支持大图上传 |

### 4. 新增: 搜索排序需求 (4 级优先)

**v5.0 状态**: 排序规则为 P0 > P1 > P2，同层级按相似度降序。

**v5.1 新增**:

搜索结果排序细化为 4 级优先级:

| 优先级 | 匹配类型 | 排序说明 |
|--------|---------|---------|
| **第 1 级** | 原图相似 | 相似度 > 0.90，与查询图高度一致 |
| **第 2 级** | 型号一致 | 同款商品，不同角度/颜色 |
| **第 3 级** | 系列相同 | 同系列产品，设计语言一致 |
| **第 4 级** | 颜色材质 | 颜色、材质等属性相近 |

排序规则: 第 1 级永远排最前，依次降级。同级别内按综合相似度降序。

## 一、项目概述

### 1.1 项目背景

SZWEGO 是服务批发行业的 B2B SaaS 电商平台，当前拥有近 40 万 VIP 商家会员，业务部门预期 2028 年发展到 100 万会员。商家经营高度专注于特定行业（服装鞋帽箱包、珠宝玉器等），典型商家约 2,500 个单品、约 1 万张商品图片。

为提升商家日常经营效率，构建"以图搜商品"核心能力，支撑两大高频业务场景：**图搜开单**（客户发图即刻找到商品发货）和**拿图找货**（在上游供应商中快速定位货源）。

### 1.2 业务目标

| 目标 | 指标 | 说明 |
|------|------|------|
| 搜索覆盖率 | ≥ 98% | 有结果返回的请求占比 |
| 搜索 CTR | ≥ 8% | 结果被点击的比例 |
| 询盘转化率 | ≥ 8% | 点击后发起询盘 |
| 图搜开单准确率 | ≥ 95% | 场景 1 首选结果即为目标商品 |
| 图搜 GMV 占比 | ≥ 3% | 通过图搜产生的交易额 |

### 1.3 核心场景

| 场景 | 用户行为 | 期望结果 | 典型用户 |
|------|---------|---------|---------|
| **图搜开单** | 客户发图，操作员找到商品开发货单 | 精准命中自家商品，2 秒内完成 | 商家操作员 |
| **拿图找货** | 拿图在关注的上家中找同款/相似货源 | 在 10~3000+ 家上家范围内找到商品 | 采购商、代购 |
| **批量查询** | 一次上传至多 7 张图，批量搜索 | 流式逐张返回，首张结果尽快到达 | 批量找货、快速比对 |
| **标签搜图** | 输入多个标签筛选商品 | 按标签命中数量排序，越精准越靠前 | 分类选品、精确筛选 |

### 1.4 非目标声明

- 不保证 100% 找到同款 — 受库内覆盖度和模型能力约束
- 不支持主观审美描述搜索（"像 Dior 那样的感觉"）
- 不替代关键词搜索 — 图搜与文本搜索互补
- 不支持视频语义检索 — 仅提取关键帧静态图片特征
- 图 + 标签混合搜索为预留能力，当前版本暂不实现

### 1.5 失败影响边界

图搜系统为增强型商品发现能力，不承担订单、库存、结算、支付等交易关键路径。最坏情况下（系统完全不可用），对核心交易链路无影响。

| 品类集中度 | 高度集中，按季 2~5 品类 | — | — |

### 2.2 商家分级与存储

| 等级 | 空间上限 | 最大图片数 (250KB/张) | 典型图片数 |
|------|---------|---------------------|-----------|
| 入门 VIP | 50 GB | ~50 万张 | ~1 万张 |

**分布特征**: 极度长尾。典型商家 1 万张 vs 头部商家 800 万张，差距 1000 倍。

| 常青池上限 | ≤ 3 亿 | ≤ 3 亿 |

### 2.4 每图数据模型

每张图片至少包含:

| 数据项 | 规格 | 存储估算 |
|--------|------|---------|
| 全局向量 | 256d float32 | ~1 KB |
| 子图向量 | 128d float32 × 最多 5 个 | ~2.5 KB |
| 商家 ID | merchant_id (标量字段) | ~32 B |
| 标签 | 8~32 个整数编码 | ~128 B |
| 三级类目 | category_l1 / l2 / l3 | ~64 B |
| URI | ≤ 128 字符 | ~128 B |
| SPU ID | product_id | ~32 B |

### 2.5 增量数据与查询量

**写入量**:

| 指标 | 当前 (2026) |
|------|-----------|
| 月增量 | ~1.4 亿图 |

| 指标 | 当前 (2026) | 放开业务限制后 (预期) | 增长倍数 |

## 三、业务特征

### 3.1 行业特征

1. **品类高度集中** — 服装鞋帽箱包商家按季节关注 2~5 个品类；珠宝玉器商家几乎单品类经营
2. **上下游同行业** — 商家关注的上家都是同行业供应商，搜索空间天然在向量空间中内聚
3. **图片长尾分布** — 绝大多数商家 ~1 万张图，极少数头部商家可达百万级

### 3.2 商家关联模型

- 每图关联 1 个商家 (merchant_id)
- 商家间通过"关注"建立上下游关系
- 单个商家关注的上家: 10~3000+ 家不等
- 商家关联为正向关系，图片属于商家

### 3.3 Query 图变形情况

用户上传的查询图片可能存在以下变形，**系统均须命中**:

| 变形类型 | 说明 | 难度 |
|---------|------|------|
| 原图缩放/有损压缩 | 图片经过转发、压缩 | 低 |
| 截取局部 | 只包含商品某个部分 | 中 |
| 原图拉伸 | 比例变形 | 中 |
| 翻拍（不同角度） | 对着屏幕拍照 | 高 |
| 原图中的子目标 | 模特穿上衣+裤子，query 只有裤子 | 高 |
| 局部翻拍+遮挡 | 拍照只拍一部分，有反光/阴影遮盖 | 最高 |

**鲁棒性基准: 人眼可识别 = 系统应可召回。**

## 四、功能需求

### 4.1 以图搜图

#### 4.1.1 场景 1: 图搜开单 (商家内搜索)

| 项目 | 要求 |
|------|------|
| 搜索范围 | 本商家商品库 (~1 万张) |
| 输入 | 1 张查询图片 |
| 输出 | 按匹配层级排序的商品列表，按 SPU 聚合 |
| 端到端延迟 | **≤ 2 秒** |
| 精度 | **Top-1 必须准确** — 发错货是业务事故 |
| 流量占比 | ~50% |

#### 4.1.2 场景 2: 拿图找货 (跨商家搜索)

| 项目 | 要求 |
|------|------|
| 搜索范围 | 本商家 + 关注的上家 (10~3000+ 家) |
| 输入 | 1 张查询图片 |
| 输出 | 按匹配层级排序的商品列表，按 SPU 聚合 |
| 精度优先级 | **找得到 + 够精准**，速度可放宽 |
| 流量占比 | ~50% |

#### 4.1.3 场景 3: 批量查询

| 项目 | 要求 |
|------|------|
| 搜索范围 | 同场景 1 或 2 |
| 输入 | ≤ 7 张查询图片 |
| 输出 | **流式逐张返回** (SSE)，先完成先推送 |
| 速度 | 首张结果尽快反馈 |
| 精度 | 准确 |

### 4.2 以标签搜图

| 项目 | 要求 |
|------|------|
| 输入 | 多个标签 |
| 匹配逻辑 | 标签交集，命中越多排越前 |
| 搜索范围 | 同样受 merchant_id 范围约束 (场景 1/2/3 均适用) |

### 4.3 图+标签混合搜索 (预留)

上传图片同时附加标签缩小范围。当前版本不实现，架构需预留扩展能力。

### 4.4 结果匹配层级

搜索结果必须按以下优先级分层呈现:

| 优先级 | 匹配类型 | 典型相似度 | 业务含义 | confidence |
|--------|---------|----------|---------|-----------|
| **P0** | 原图匹配 | > 0.90 | 可直接开单 | HIGH |
| **P1** | 款式匹配 (同款不同色) | 0.70~0.90 | 需确认后开单 | MEDIUM |
| **P2** | 品类 + 风格一致 | 0.50~0.70 | 仅供参考选品 | LOW |

**排序规则**: P0 永远排最前 > P1 > P2。同层级内按相似度降序。

**confidence 语义映射**:

| confidence | 前端提示 |
|-----------|---------|
| HIGH | "找到了与图片高度一致的商品" |
| MEDIUM | "未找到完全同款，以下是相似款式推荐" |
| LOW | "暂未找到高度相似商品，以下是同类推荐" |

### 4.5 结果聚合

按 **SPU (product_id) 聚合**: 同一单品的多张图片 (不同角度/场景) 命中任一张，都应归到同一商品展示，不重复展示。

### 4.6 搜索范围过滤

| 过滤维度 | 实现方式 | 说明 |
|---------|---------|------|
| **merchant_id** | Milvus 标量预过滤 (非后过滤) | 搜索前缩小范围，保证召回率 |
| time_range | 分区路由 (热区/非热区/常青) | 5m / 18m / all |
| data_scope | 分区路由 | evergreen / rolling / all |

**merchant_id 预过滤为强制要求**: 单商家 ~1 万张图在 30 亿总量中占比 0.00025%，全局 ANN + 后过滤召回率趋近于零。必须在 ANN 搜索过程中通过 Milvus 标量表达式前置过滤。

## 五、检索策略

### 5.1 搜索原则

1. **保证召回率和准确性为主，调用执行时间次之**
2. **能在调用前缩小范围就先缩小范围**

### 5.2 检索管线

```
Stage 0  : merchant_scope 解析 + 降级状态覆盖
           ↓ (并行)
Stage 1  : 特征提取 (全局向量 + 子图向量 + 标签 + 类目预测)
           bitmap 预构建 (与特征提取并行)
           ↓
Stage 1.5+2-H : 标签召回 + 热区 ANN **并行执行**
           merchant_id 预过滤写入 Milvus expr
           类目预过滤写入 Milvus expr (高置信度时)
           ↓
Stage 2-NH : [级联] 非热区 DiskANN (Top1 score < 0.65 时触发)
           ↓
Stage 3  : 候选合并去重
           ↓
Stage 4  : Refine 精排 (float32 重计算)
           ↓
Stage 5-7: [条件] Fallback 子图 + 标签多路召回
           ↓
Stage 8  : 按 SPU 聚合 → 匹配层级分类 → 响应构建
```

### 5.3 双路搜索 (必备能力)

**全局向量 + 子图向量双路搜索是必备能力，非可选 fallback。**

| 路径 | 覆盖场景 | 说明 |
|------|---------|------|
| 全局向量 ANN | 原图、缩放、压缩、翻拍 | 主召回路径 |
| 子图向量 ANN | 截取局部、子目标、局部翻拍+遮挡 | 不可或缺的补充路径 |

两路并行执行，结果融合排序。在商家内 ~1 万张的小子集上，双路并行代价极低。

### 5.4 融合排序

```
final_score = 0.40 × global_sim        (全局向量相似度)
            + 0.30 × max(sub_sims)     (子图向量最高相似度)
            + 0.20 × tag_overlap       (标签重合度)
            + 0.10 × cat_match         (类目匹配)
```

### 5.5 前置缩小范围策略

| 策略 | 时机 | 效果 |
|------|------|------|
| merchant_id 标量预过滤 | ANN 搜索中 | 30 亿 → 1 万~3000 万 |
| 类目预过滤 (高置信度 ≥85%) | ANN 搜索中 | 进一步缩小搜索空间 |
| bitmap 预构建与特征提取并行 | Stage 0~1 | 消除串行等待 |
| 标签召回与 ANN 并行 | Stage 1.5+2-H | 节省 10~20ms |
| 商家范围自适应 top_k | ANN 搜索中 | 有过滤时放大候选池 ×1.5 |

## 六、写入需求

### 6.1 写入链路

每张商品图入库时必须完成:

```
原图 → 全局向量提取 (256d float32)
    → 目标检测 (上衣/裤子/包/鞋等)
       → 各子图向量提取 (128d float32, 最多 5 个)
    → 标签提取 (8~32 个)
    → 类目推导 (三级类目)
    → Milvus 入库 (向量 + merchant_id + 标签 + 类目 + product_id)
    → 可被查询命中
```

### 6.2 写入可见性

**用户上新后 ≤ 3 分钟端到端可被搜索到。**

单次上新最多 9 张图，3 分钟时间预算拆解:

| 环节 | 耗时 | 说明 |
|------|------|------|
| 手机上传 | 3~10 秒/张 × 9 张 = 27~90 秒 | 客户端网络 |
| 合规审查 | 5 秒/张 × 9 张 = 45 秒 | 内容安全 |
| **入库写入** | **≤ 45 秒 (9 张合计)** | 全链路处理 |

单张入库预算 ≤ 5 秒，含: 全局向量提取 → 子图检测 + 向量提取 → 标签提取 → Milvus 入库 → 索引生效 → 可查询。

### 6.3 子图检测要求

| 规则 | 说明 |
|------|------|
| 面积阈值 | 子图面积 < 原图 10% 时跳过 |
| 数量上限 | 每图最多 5 个子图 (按 confidence 降序 Top5) |
| 写入时机 | 写入侧全量检测提取，非查询侧延迟提取 |

## 八、高可用与降级

### 8.1 可用性目标

| 功能层 | SLA | 月停机上限 |
|--------|-----|----------|
| 核心功能 (热区 + 常青) | 99.99% | ≤ 4.5 分钟 |
| 完整功能 (全量 + 常青) | 99.5% | ≤ 3.9 小时 |

### 8.2 降级策略

| 状态 | 触发条件 | 行为 |
|------|---------|------|
| **S0 正常** | 默认 | 全功能 |
| **S1 预降级** | P99 > 600ms 持续 3min OR 错误率 > 0.1% | 禁用非热区级联；保留 fallback 多路召回 |
| **S2 强降级** | P99 > 1000ms 持续 1min OR 错误率 > 2% | 仅热区 + 常青；禁用 fallback |
| **S3 恢复中** | S1/S2 恢复后 | 灰度放量 10% → 50% → 100% |
| **S4 人工锁定** | 管理员手动 | 自定义参数 |

**降级原则**: 召回率优先 — S1 保留 fallback 保证多路召回能力，仅 S2 才全面裁剪。

### 8.3 商家过滤降级

Milvus 标量预过滤为主路径，bitmap 后过滤为备用:

```
Level 0: Milvus merchant_id expr 标量预过滤 (主路径)
Level 1: gRPC bitmap-filter-service 后过滤 (备用)
Level 2: PG 直查 rb_intersect (兜底)
Level 3: 跳过过滤 (filter_skipped=true)
```

## 十、架构关键约束

| 约束 | 原因 | 优先级 |
|------|------|--------|
| `merchant_id` Milvus 标量预过滤 | 后过滤召回率 ≈ 0%，架构不可行 | **P0 必须** |
| 全局 + 子图双路并行搜索 | 截取/子目标/局部翻拍场景为刚需 | **P0 必须** |
| 写入时全量子图检测 | 查询时不能依赖延迟提取 | **P0 必须** |
| 按 SPU 聚合返回 | 同一商品多图不重复展示 | **P1 重要** |
| 场景 3 SSE 流式返回 | 首张结果尽快反馈 | **P1 重要** |
| 图+标签混合搜索预留 | 未来扩展 | **P2 预留** |
