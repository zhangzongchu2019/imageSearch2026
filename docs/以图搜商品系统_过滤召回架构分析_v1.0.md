# 以图搜商品系统 — 过滤与召回架构分析

> 版本: v2.0
> 日期: 2026-03-31
> 状态: 分析结论

---

## 一、问题背景

系统需要在 30 亿向量中按商家范围过滤搜索。核心挑战:

1. 单商家图片量 ~1 万张，占全局 0.00025%，选择率极低
2. **图片-商家是多对多关系**: 商家可转发保存其他商家的产品，一张图可属于多个商家
3. 需要在保证召回率的前提下，找到可行的过滤方案

### 1.1 关键业务数据

| 指标 | 值 |
|------|-----|
| 每日新上传产品 | 80 万 |
| 每日新上传照片 | 400 万张 |
| 每日商品记录总量 (含转发) | **2000 万** |
| 每日转发量 | 2000万 - 80万 = **1920 万** |
| **转发倍率** | **24 倍** (每产品平均被转发 24 次) |
| 每张图平均关联商家数 | ~25 个 |

---

## 二、图片-商家多对多关系

### 2.1 业务模型

```
商家 A 上传产品 (含 5 张图)
  → 5 张图属于 A

商家 B 转发保存 A 的产品
  → 同 5 张图, 现在同时属于 A 和 B

商家 C、D、E... 也转发
  → 同 5 张图, 属于 A、B、C、D、E...
  → 一张图可关联数十到数千个商家
```

### 2.2 对 merchant_id 单值方案的否决

| 方案 | 问题 | 结论 |
|------|------|------|
| merchant_id 单值标量字段 | 只能表达 1:1, 无法表达转发后的 1:N | **不可行** |
| 向量记录复制 (每商家一份) | 存储膨胀 25 倍 (30亿→750亿), 写入量 25 倍 | **不可行** |
| Milvus 数组字段 merchant_ids[] | 日均近 1 亿次 upsert (转发), Milvus 扛不住 | **不可行** |

**结论: bitmap 是唯一能高效处理 1:N 多对多关系 + 高频转发更新的方案。bitmap 基础设施必须保留。**

## 三、商家规模与 HNSW 选择率

### 3.1 商家规模分布

| 商家类型 | 图片量 | 占全局 30 亿 | 占比商家数 |
|---------|--------|------------|----------|
| 典型商家 | ~1 万 | 0.00025% | 99%+ |
| 中等商家 | ~10 万 | 0.003% | 少量 |
| 大商家 | ~50 万 | 0.017% | 极少 |
| 头部商家 | ~800 万 | 0.27% | 极少数 |

### 3.2 HNSW filtered ANN 的选择率问题

全局 ANN + bitmap 后过滤的召回率:

| 场景 | ANN top_k=3000 期望命中 | 召回率 |
|------|----------------------|--------|
| 单商家 1 万图 | 3000 × 0.00025% ≈ 0.008 | ≈ 0% |
| 关注 100 家 | 3000 × 0.03% ≈ 0.9 | 极低 |
| 关注 3000 家 | 3000 × 1% ≈ 30 | 可用 |

**结论: 全局 ANN + 后过滤对小范围搜索 (单商家、少量关注) 召回率趋近于零。**

## 四、分区方案分析（已否决）

### 4.1 仅按月分区 (19 分区)

每月分区 ~1.67 亿条，单商家选择率 0.00033%。不解决问题。

### 4.2 月分区 × 哈希子分区 (304 分区)

16 哈希后选择率提升到 0.005%，仍不够。管理复杂度高 (304 分区)，99%+ 商家不受益。

### 4.3 为什么纯分区方案走不通

要让 HNSW 对典型商家高效工作，需要选择率 > 5%，分区数需 27 万+，远超 Milvus 上限。

## 五、最终方案: 标准月分区 + SVIP 月分区 + bitmap 双角色

### 5.1 核心思路

- **bitmap 保留，但角色升级**: 从"ANN 后过滤"升级为"前置预查 pk 列表 + 后置精确过滤"
- **小范围搜索 (99%+ 流量)**: bitmap 预查 pk 列表 → Milvus 按 pk 取向量 → 暴力搜索
- **大范围搜索**: Milvus ANN 粗召回 → bitmap 后过滤
- **SVIP 大商家**: 独立分区提升后过滤选择率

### 5.2 数据分区

```
Milvus Collection: global_images

标准月分区 (所有商家的原始图片):
├── month_202601  (热区, HNSW)
├── month_202602  (热区, HNSW)
├── ...
├── month_202606  (热区, HNSW)     ← 热区 6 个月
├── month_202507  (冷区, DiskANN)
├── ...
├── month_202412  (冷区, DiskANN)  ← 冷区 12 个月
├── month_evergreen                 ← 常青
│                                   小计: 19 个分区

SVIP 月分区 (≥ 1000 万图的头部商家, ~16 家共享):
├── svip_202601   (热区, HNSW)
├── ...
├── svip_202606   (热区, HNSW)
├── svip_202507   (冷区, DiskANN)
├── ...
├── svip_202412   (冷区, DiskANN)
├── svip_evergreen
│                                   小计: 19 个分区

合计: 38 个分区

注意: Milvus 不存储 merchant 信息, 仅存 image_pk + 向量 + 标签 + 类目
商家关联关系由 bitmap 体系维护 (PG + RocksDB + Flink + CDC)
```

### 5.3 自适应搜索策略

```
                    商家范围大小
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
     小范围 (≤100家)  中范围        大范围 (>1000家)
     含单商家场景1   (100~1000家)    场景2大范围
            │            │            │
            ▼            ▼            ▼
     bitmap 预查     bitmap 预查    Milvus ANN
     image_pk 列表   image_pk 列表  粗召回
     (≤100万条)      (100万~1000万)  (top_k放大)
            │            │            │
            ▼            ▼            ▼
     Milvus 按 pk    Milvus ANN     bitmap
     取向量           + bitmap       后过滤
     + 暴力搜索       后过滤
            │            │            │
            ▼            ▼            ▼
        < 10ms        50~150ms     100~300ms
        精度100%       精度~98%     精度~95%
```

### 5.4 场景 1: 单商家搜索 (图搜开单)

**bitmap 前置预查 + 暴力搜索, 精度 100%**

```python
async def search_single_merchant(merchant_id, query_vec):
    # Step 1: 从 bitmap (RocksDB) 预查商家的所有 image_pk
    # RocksDB 本地查询, ~2ms
    image_pks = await bitmap_service.get_merchant_image_pks(merchant_id)
    # 典型商家 ~1 万条

    # Step 2: Milvus 按 pk 批量取向量 (或从缓存)
    # ~5ms (走缓存则 ~1ms)
    vectors = await milvus.query(
        expr=f"image_pk in {image_pks}",
        output_fields=["image_pk", "global_vec", "product_id", "tags", ...],
        partition_names=get_hot_partitions(),  # 仅热区+常青
    )

    # Step 3: 进程内暴力计算 cosine distance
    # ~1ms (1万条 × 256d)
    results = brute_force_cosine(query_vec, vectors)

    return results  # 精度 100%, 合计 ~8ms
```

### 5.5 场景 2: 跨商家搜索 (拿图找货)

**按范围大小选择策略:**

```python
async def search_cross_merchant(merchant_ids, query_vec):
    # Step 1: 从 bitmap 预查所有目标商家的 image_pk 总量
    total_pks = await bitmap_service.get_total_count(merchant_ids)

    if total_pks <= BRUTE_FORCE_LIMIT:  # e.g. 100 万
        # 小/中范围: bitmap 预查 + 暴力搜索 (同场景 1)
        image_pks = await bitmap_service.get_merchant_image_pks_batch(merchant_ids)
        vectors = await milvus.query(expr=f"image_pk in {image_pks}", ...)
        return brute_force_cosine(query_vec, vectors)
    else:
        # 大范围: ANN 粗召回 + bitmap 后过滤
        ann_results = await milvus.search(
            data=[query_vec],
            param={"metric_type": "COSINE", "params": {"ef": 256}},
            limit=top_k * 3,  # 放大候选池, 补偿后过滤损耗
        )
        # bitmap 后过滤: 只保留属于目标商家的
        filtered = await bitmap_service.filter(ann_results, merchant_ids)
        return filtered
```

### 5.6 SVIP 大商家搜索

SVIP 数据在独立分区内，ANN + bitmap 后过滤的选择率大幅提升:

```
SVIP 分区内: ~16 个 SVIP 共享
单 SVIP 选择率: ~6.25% (vs 全局 0.27%)
→ ANN + bitmap 后过滤在 SVIP 分区内高效工作
```

```python
async def search_svip_merchant(merchant_id, query_vec):
    # SVIP 分区内 ANN, 选择率 ~6.25%
    ann_results = await milvus.search(
        data=[query_vec],
        partition_names=get_svip_partitions(merchant_id),
        param={"metric_type": "COSINE", "params": {"ef": 256}},
        limit=top_k * 2,  # 适度放大
    )
    # bitmap 后过滤
    filtered = await bitmap_service.filter(ann_results, [merchant_id])
    return filtered
```

## 六、SVIP 路由表

### 6.1 核心规则

**同一个 merchant_id 在同一个月份，只能存在于一种分区中 (standard 或 svip)，不能两边都有。**

### 6.2 路由表 (PostgreSQL)

```sql
CREATE TABLE svip_partition_roster (
    merchant_id   VARCHAR(64)  NOT NULL,
    ts_month      INT          NOT NULL,   -- 202601
    partition_type VARCHAR(10)  NOT NULL,   -- 'svip' | 'standard'
    svip_pool_id  INT,                      -- SVIP 池编号 (partition_type='svip' 时)
    created_at    TIMESTAMP    DEFAULT now(),
    PRIMARY KEY (merchant_id, ts_month)
);

CREATE INDEX idx_roster_month_type ON svip_partition_roster(ts_month, partition_type);
```

路由查询缓存在 Redis, key: `route:{merchant_id}:{ts_month}`, TTL 1 小时。

### 6.3 写入路由

```python
async def get_write_partition(merchant_id, ts_month):
    route = await cache.get(f"route:{merchant_id}:{ts_month}")
    if route:
        return route

    row = await pg.fetchone(
        "SELECT partition_type FROM svip_partition_roster "
        "WHERE merchant_id=$1 AND ts_month=$2", merchant_id, ts_month
    )

    if row is None:
        partition = f"month_{ts_month}"
        await pg.execute(
            "INSERT INTO svip_partition_roster VALUES ($1,$2,'standard',NULL)",
            merchant_id, ts_month
        )
    elif row['partition_type'] == 'svip':
        partition = f"svip_{ts_month}"
    else:
        partition = f"month_{ts_month}"

    await cache.set(f"route:{merchant_id}:{ts_month}", partition, ex=3600)
    return partition
```

### 6.4 搜索路由

搜索时按路由表逐月判断每个商家的数据在哪个分区:

```python
async def get_search_partitions(merchant_id, time_range_months):
    routes = await get_routes(merchant_id, time_range_months)

    standard_months = []
    svip_months = []
    for ts_month, ptype in routes:
        if ptype == 'svip':
            svip_months.append(f"svip_{ts_month}")
        else:
            standard_months.append(f"month_{ts_month}")

    return standard_months, svip_months
```

### 6.5 SVIP 晋升与降级

**从下个月开始生效，历史月份不动，无需数据迁移。**

| 事件 | 触发条件 | 操作 |
|------|---------|------|
| 晋升 | 商家图片量跨过 1000 万 | 路由表: 下月起 partition_type='svip' |
| 降级 | 商家图片量降至 500 万以下 | 路由表: 下月起 partition_type='standard' |
| SVIP 池扩容 | 已有 SVIP 分区容纳 >16 家 | 新增一组 svip 月分区 (38 → 57) |

```
晋升示例:
  商家 M123 在 202606 达到 1000 万图
  → 路由表: (M123, 202607, 'svip')
  → 202607 起新数据写入 svip_202607
  → 历史数据留原分区, 18 月后自然 drop

降级示例:
  商家 M123 在 202609 降至 500 万以下
  → 路由表: (M123, 202610, 'standard')
  → 搜索时按路由表逐月判断, 混合查询两种分区
```

**关键优势: 不需要数据迁移。** 历史数据留原分区, 18 个月后自然过期 drop。

## 七、bitmap 体系的角色定义

### 7.1 bitmap 的双重角色

| 角色 | 使用场景 | 说明 |
|------|---------|------|
| **前置预查** | 小范围搜索 (≤100 万 pk) | 先查 pk 列表, 再按 pk 取向量暴力搜索 |
| **后置过滤** | 大范围搜索 (>100 万 pk) | ANN 粗召回后, bitmap 交集过滤 |

### 7.2 bitmap 基础设施 (全保留)

| 组件 | 角色 | 保留原因 |
|------|------|---------|
| **PG merchant_id_mapping** | 商家 ID → bitmap_index 编码 | 转发写入主路径 |
| **PG image_merchant_bitmaps** | image_pk → 商家 Roaring Bitmap | 多对多关系存储 |
| **Kafka merchant-events** | 转发事件流 | 日均 1920 万转发 |
| **Flink pipeline** | 事件聚合, bitmap 合并 | 窗口批量处理 |
| **RocksDB (bitmap-filter-service)** | 本地 bitmap 缓存 | 搜索节点本地查询 ~2ms |
| **CDC Kafka → RocksDB** | bitmap 同步 | 保持本地缓存一致 |

### 7.3 bitmap 新增能力: 按商家查 pk 列表

当前 bitmap 支持: 给定 candidate_pks + merchant_scope → 过滤后 pks (后过滤)

新增需要支持: 给定 merchant_id → 返回该商家的所有 image_pk 列表 (前置预查)

```
现有: filter(candidate_pks, merchant_scope) → filtered_pks
新增: get_image_pks(merchant_id) → [pk1, pk2, ..., pkN]
```

实现方式: RocksDB 新增反向索引 `merchant_id → image_pk_list`，或从 PG 查询。

## 八、暴力搜索的数据传输优化

### 8.1 数据量

```
1 万条 × 256d × 4 bytes = ~10 MB (全局向量)
1 万条 × 3 子图 × 128d × 4 bytes = ~15 MB (子图向量)
单次查询传输: ~25 MB
```

### 8.2 缓存方案

商家图搜开单是高频重复场景，同一商家一天搜几十上百次:

```
L1 进程内缓存: 热门 500 家 → ~12.5 GB
L2 Redis 缓存: 活跃 5000 家 → ~125 GB (Redis 集群)
L3 Milvus 标量查询: 缓存未命中时回源

缓存 TTL: 5 分钟 (写入可见性 ≤ 3 分钟, 留裕量)
```

### 8.3 Milvus 自动退化验证

Milvus 2.5 在过滤后候选数量极少时 (< 总量 ~0.1%)，可能自动退化为 brute-force。如果验证可行，则无需传输向量到进程内:

```python
# 验证测试: Milvus filtered search 在极低选择率下的表现
results = collection.search(
    data=[query_vec],
    anns_field="global_vec",
    param={"metric_type": "COSINE", "params": {"ef": 256}},
    expr="image_pk in [pk1, pk2, ...]",  # 预查到的 pk 列表
    limit=100,
)
# 如果延迟 < 50ms 且召回完整 → 直接用 Milvus, 无需外置暴力搜索
# 如果延迟不可接受 → 启用缓存 + 进程内暴力搜索方案
```

## 九、月分区生命周期管理

两套分区 (标准 + SVIP) 采用完全相同的管理策略:

| 操作 | 频率 | 标准分区 | SVIP 分区 |
|------|------|---------|----------|
| 新建分区 | 每月 | month_YYYYMM | svip_YYYYMM |
| 冷热切换 | 每月 | 第 6→7 月 HNSW→DiskANN | 同 |
| TTL 清理 | 每月 | drop 过期月分区 | drop 过期月分区 |
| 索引构建 | 每月 | 新分区 build index | 同 |
| 内存管理 | 每月 | 热区 load, 冷区 release | 同 |

**一套 cron-scheduler 逻辑，操作两组分区名前缀 (month_ / svip_)。**

## 十、总结

### 10.1 关键发现

| 发现 | 影响 |
|------|------|
| 图片-商家是多对多 (转发倍率 24x) | 向量复制和数组字段不可行, bitmap 必须保留 |
| merchant_id 不能作为 Milvus 标量字段做预过滤 | 搜索预过滤改为 bitmap 预查 pk 列表 |
| 小范围搜索 (99%+ 流量) 选择率极低 | bitmap 预查 + 暴力搜索是唯一可行方案 |
| 大范围搜索选择率可接受 | ANN + bitmap 后过滤可用 |
| SVIP 分区提升后过滤选择率 | 大商家从 0.27% 提升到 6.25% |

### 10.2 架构决策

| 决策 | 方案 | 原因 |
|------|------|------|
| 数据分区 | 标准月分区 19 + SVIP 月分区 19 = **38 个分区** | 简单统一, 同一套生命周期管理 |
| bitmap 体系 | **全保留**, 角色升级为前置预查 + 后置过滤 | 多对多关系 + 高频转发, 唯一可行方案 |
| 小范围搜索 | bitmap 预查 pk → Milvus 取向量 → 暴力搜索 | 精度 100%, < 10ms |
| 大范围搜索 | Milvus ANN 粗召回 → bitmap 后过滤 | 选择率够高时可用 |
| SVIP 搜索 | SVIP 分区内 ANN + bitmap 后过滤 (选择率 ~6.25%) | 30~50ms |
| SVIP 路由 | PG 路由表 svip_partition_roster | 唯一真相, 按月生效, 无需迁移 |
| SVIP 阈值 | 商家图片量 ≥ 1000 万 | 每 SVIP 分区容纳 ~16 个 SVIP |
| TTL 清理 | drop 月分区 | 两套分区统一操作, 无 delete 问题 |
| 向量缓存 | L1 进程 / L2 Redis | 减少暴力搜索场景 Milvus 数据传输 |
