# 以图搜商品系统 — 过滤与召回架构分析

> 版本: v1.2

> 状态: 分析结论

---

## 一、问题背景

系统需要在 30 亿向量中按 merchant_id 过滤搜索。单商家典型图片量 ~1 万张，占全局 0.00025%。需要分析在不同过滤选择率下，Milvus 检索方案的可行性，以及对系统架构的影响。

---

## 二、核心矛盾

### 2.1 商家规模分布

| 商家类型 | 图片量 | 占全局 30 亿 | 占比商家数 |
|---------|--------|------------|----------|
| 典型商家 | ~1 万 | 0.00025% | 99%+ |
| 中等商家 | ~10 万 | 0.003% | 少量 |
| 大商家 | ~50 万 | 0.017% | 极少 |
| 头部商家 | ~800 万 | 0.27% | 极少数 |

### 2.2 HNSW filtered ANN 的局限

Milvus 的 HNSW filtered ANN 在图遍历过程中检查标量条件。当选择率极低时：

```
HNSW 遍历节点 → 检查 merchant_id → 不匹配 → 继续遍历
```

- 选择率 0.00025% (典型商家): 需遍历约 400 万节点才能找到 ~10 条匹配, **不可行**
- 选择率 > 5%: HNSW 正常高效工作

**小商家暴力搜索够快但 HNSW 过滤不行，大商家 HNSW 过滤能用但暴力搜索不行。**

## 三、分区方案分析（已否决）

### 3.1 仅按月分区 (19 分区)

```
18 个月 + 常青 = 19 个分区
每月: 30 亿 / 18 ≈ 1.67 亿/分区
```

| 商家类型 | 每月图片 | 分区内选择率 | HNSW 可行？ |
|---------|---------|------------|-----------|
| ~1 万 | 556 | 0.00033% | 不可行 |
| ~800 万 | 44 万 | 0.27% | 边界 |

**结论**: 月分区对数据生命周期管理有价值 (冷热分离、TTL 清理)，但不解决 merchant_id 过滤问题。

### 3.2 月分区 × 哈希子分区（已否决）

| 子分区数 | 总分区数 | 每子分区大小 | 典型商家选择率 | 头部商家选择率 |
|---------|---------|------------|-------------|-------------|
| 16 | 304 | 1044 万 | 0.005% | 4.3% |
| 64 | 1,153 | 261 万 | 0.021% | 17% |
| 128 | 2,305 | 130 万 | 0.043% | 34% |
| 256 | 4,609 | — | — | 超 Milvus 4096 上限 |

**否决原因**: 304 个分区管理复杂度高 (每月运维 ×16)，而 99%+ 商家走标量查询 + 暴力搜索根本不经过 HNSW，仅 <1% 头部商家受益。收益不值得复杂度。

### 3.3 SVIP 独占分区（已否决）

每个 SVIP 商家独占 1 个分区。

**否决原因**: 200~500 个 SVIP 分区，管理负担大，且 SVIP 分区若不按月拆分则无法通过 drop 分区清理过期数据。Milvus delete 是标记删除 + compaction，大批量删除性能代价高。

### 3.4 为什么纯分区方案走不通

要让 HNSW filtered ANN 对典型商家 (556 条/月) 高效工作，选择率需 > 5%:

```
每分区最大容量 = 556 / 5% = 11,120 条
需要分区数 = 1.67 亿 / 11,120 ≈ 15,000 个/月
总分区数 = 18 × 15,000 = 270,000 个 → 远超 Milvus 上限
```

## 四、最终方案：标准月分区 + SVIP 月分区 + 自适应搜索

### 4.1 核心思路

- **普通商家 (< 1000 万图)**: 数据存入标准月分区，搜索时标量查询 + 暴力搜索
- **SVIP 商家 (≥ 1000 万图)**: 数据存入 SVIP 月分区 (多个 SVIP 共享)，搜索时 HNSW filtered ANN
- **两套分区采用完全相同的月度生命周期管理**，统一运维

### 4.2 SVIP 共享分区的容量设计

```
1 个月分区容量 = ~1.67 亿条
1 个 SVIP 每月图片 = ~1000 万 / 18 ≈ 55 万条
1 个 SVIP 月分区可容纳 = 1.67 亿 / 55 万 ≈ 16 个 SVIP

单 SVIP 搜索选择率 = 55 万 / (16 × 55 万) = 6.25% → HNSW 高效工作 ✓
```

### 4.3 分区布局

```
Milvus Collection: global_images

标准月分区 (普通商家, < 1000 万图):
├── month_202601  (热区, HNSW)
├── month_202602  (热区, HNSW)
├── ...
├── month_202606  (热区, HNSW)     ← 热区 6 个月
├── month_202507  (冷区, DiskANN)
├── ...
├── month_202412  (冷区, DiskANN)  ← 冷区 12 个月
├── month_evergreen                 ← 常青
│                                   小计: 19 个分区

SVIP 月分区 (≥ 1000 万图, ~16 个 SVIP 共享):
├── svip_202601   (热区, HNSW)
├── svip_202602   (热区, HNSW)
├── ...
├── svip_202606   (热区, HNSW)     ← 热区 6 个月
├── svip_202507   (冷区, DiskANN)
├── ...
├── svip_202412   (冷区, DiskANN)  ← 冷区 12 个月
├── svip_evergreen                  ← 常青
│                                   小计: 19 个分区

合计: 19 + 19 = 38 个分区

标量索引: merchant_id (INVERTED), category_l1 (INVERTED), tags (INVERTED)
```

### 4.4 方案对比

| | ×16 哈希 (304 分区) | SVIP 独占 (219~519) | **标准+SVIP 月分区 (38)** |
|---|---|---|---|
| 分区总数 | 304 | 219~519 | **38** |
| 管理复杂度 | 高 | 中 | **低** |
| 运维逻辑 | 每月 ×16 操作 | 两套不同逻辑 | **一套逻辑管两套分区** |
| TTL 清理 | drop ×16 | 需 delete + compaction | **drop 分区，干净利落** |
| 冷热分离 | 天然 | SVIP 无冷热 | **两套都天然分离** |
| SVIP 搜索选择率 | 4.3% | 100% | **6.25%** |
| SVIP 搜索延迟 | 50~80ms | < 20ms | **30~50ms** |
| 普通商家影响 | 每月 ×16 运维 | 零 | **零** |

### 4.5 自适应搜索策略

```
                    merchant_scope 判定
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
        普通商家      SVIP 商家     混合 (跨商家)
       (标准分区)     (SVIP分区)    (两套并行)
            │            │            │
            ▼            ▼            ▼
       标量查询      HNSW filtered  分别查询

            ▼            │            ▼
         < 10ms       30~50ms      合并排序
         精度100%      精度~98%
```

### 4.6 SVIP 路由表

**核心规则: 同一个 merchant_id 在同一个月份，只能存在于一种分区中 (standard 或 svip)，不能两边都有。**

路由表是分区归属的唯一真相来源。

```sql
CREATE TABLE svip_partition_roster (
    merchant_id   VARCHAR(64)  NOT NULL,
    ts_month      INT          NOT NULL,   -- 202601
    partition_type VARCHAR(10)  NOT NULL,   -- 'svip' | 'standard'
    svip_pool_id  INT,                      -- SVIP 池编号 (partition_type='svip' 时)
    created_at    TIMESTAMP    DEFAULT now(),
    PRIMARY KEY (merchant_id, ts_month)
);

-- 索引: 按月查询某个池的所有 SVIP
CREATE INDEX idx_roster_month_type ON svip_partition_roster(ts_month, partition_type);
```

路由查询结果缓存在 Redis, key: `route:{merchant_id}:{ts_month}`, TTL 1 小时。

### 4.7 写入路由逻辑

```python
async def get_write_partition(merchant_id, ts_month):
    # 查路由表 (Redis 缓存优先)
    route = await cache.get(f"route:{merchant_id}:{ts_month}")
    if route:
        return route

    # 缓存未命中, 查 PG
    row = await pg.fetchone(
        "SELECT partition_type FROM svip_partition_roster "
        "WHERE merchant_id=$1 AND ts_month=$2", merchant_id, ts_month
    )

    if row is None:
        # 新商家或新月份, 默认 standard
        partition = f"month_{ts_month}"
        await pg.execute(
            "INSERT INTO svip_partition_roster VALUES ($1,$2,'standard',NULL)",
            merchant_id, ts_month
        )
    elif row['partition_type'] == 'svip':
        partition = f"svip_{ts_month}"
    else:
        partition = f"month_{ts_month}"

    await cache.set(f"route:{merchant_id}:{ts_month}", partition, ex=3600)
    return partition
```

### 4.8 搜索路由逻辑

搜索时按路由表逐月判断每个商家的数据在哪个分区，分组后并行查询:

```python
async def search(merchant_ids, query_vec, time_range_months):
    # 批量查路由表, 按分区类型分组
    standard_partitions = set()
    svip_partitions = {}  # {svip_merchant_id: [partition_names]}
    normal_merchant_ids = set()

    for mid in merchant_ids:
        routes = await get_routes(mid, time_range_months)
        for ts_month, ptype in routes:
            if ptype == 'svip':
                svip_partitions.setdefault(mid, []).append(f"svip_{ts_month}")
            else:
                standard_partitions.add(f"month_{ts_month}")
                normal_merchant_ids.add(mid)

    tasks = []

    # 普通商家/普通月份: 标量查询 + 暴力搜索
    if normal_merchant_ids:
        tasks.append(scalar_brute_force(
            list(standard_partitions), list(normal_merchant_ids), query_vec
        ))

    # SVIP 月份: HNSW filtered ANN
    for svip_id, partitions in svip_partitions.items():
        tasks.append(hnsw_filtered(partitions, svip_id, query_vec))

    results = await asyncio.gather(*tasks)
    return merge_and_rank(results)
```

### 4.9 SVIP 晋升与降级

**从下个月开始生效，历史月份不动，无需数据迁移。**

| 事件 | 触发条件 | 操作 |
|------|---------|------|
| 晋升 | 商家图片量跨过 1000 万 | 路由表: 下月起 partition_type='svip' |
| 降级 | 商家图片量降至 500 万以下 | 路由表: 下月起 partition_type='standard' |
| SVIP 池扩容 | 已有 SVIP 分区容纳 >16 家 | 新增一组 svip 月分区 (38 → 57) |

晋升示例:

```
商家 M123 在 202606 达到 1000 万图
  ↓
路由表写入: (M123, 202607, 'svip')
202607 起新数据写入 svip_202607
202606 及之前的历史数据仍在 month_YYYYMM, 不迁移
搜索时: 202607+ 查 svip 分区, 202606- 查 standard 分区
历史数据 18 个月后自然过期, drop 月分区即可清理
```

降级示例:

```
商家 M123 在 202609 降至 500 万以下
  ↓
路由表写入: (M123, 202610, 'standard')
202610 起新数据写入 month_202610
202607~202609 数据仍在 svip_YYYYMM, 不迁移
搜索时: 按路由表逐月判断, 混合查询两种分区
```

**关键优势: 不需要数据迁移**

| | 迁移方案 | 按月生效方案 |
|---|---|---|
| 晋升/降级操作 | 搬迁历史数据, 耗时长 | **仅修改路由表, 下月生效** |
| 数据一致性 | 迁移中可能双写/漏写 | **路由表是唯一真相** |
| 历史数据 | 需搬到新分区 | **留原分区, 18 月后自然 drop** |

### 4.10 商家图片计数维护

- 写入时: 原子递增 Redis 计数 `merchant_img_count:{merchant_id}`
- TTL 过期清理时: 批量递减
- 月度评估: cron 任务检查是否触发晋升/降级阈值, 更新路由表
- 精度要求: 近似值即可，不影响功能正确性

## 五、暴力搜索的数据传输优化

### 5.1 问题

标量查询 + 暴力搜索需要从 Milvus 传输向量数据:

```
1 万条 × 256d × 4 bytes = ~10 MB (全局向量)
1 万条 × 3 子图 × 128d × 4 bytes = ~15 MB (子图向量)
单次查询传输: ~25 MB
```

### 5.2 缓存方案

商家图搜开单是高频重复场景，同一商家一天搜几十上百次:

```
L1 进程内缓存: 热门 500 家 → ~12.5 GB
L2 Redis 缓存: 活跃 5000 家 → ~125 GB (Redis 集群)
L3 Milvus 标量查询: 缓存未命中时回源

缓存 TTL: 5 分钟 (写入可见性 ≤ 3 分钟, 留裕量)
```

### 5.3 Milvus 自动退化验证

Milvus 2.5 在过滤后候选数量极少时 (< 总量 ~0.1%)，可能自动退化为 brute-force。如果验证可行，则无需传输向量到进程内:

```python
# 验证测试: Milvus filtered search 在极低选择率下的表现
results = collection.search(
    data=[query_vec],
    anns_field="global_vec",
    param={"metric_type": "COSINE", "params": {"ef": 256}},
    expr="merchant_id == 'TEST_MERCHANT'",
    limit=100,
)
# 如果延迟 < 50ms 且召回完整 → 直接用 Milvus, 架构最简
# 如果延迟不可接受 → 启用缓存方案
```

## 六、场景 2 跨商家搜索的过滤分析

### 6.1 选择率 (标准月分区内)

| 关注商家数 | 候选量 | 月分区选择率 | 策略 |
|-----------|--------|------------|------|
| 10 家 | ~10 万 | 0.06% | 标量 + 暴力 |
| 100 家 | ~100 万 | 0.6% | 标量 + 暴力 (边界) |
| 1000 家 | ~1000 万 | 6% | HNSW filtered (正常) |
| 3000 家 | ~3000 万 | 18% | HNSW filtered (高效) |

### 6.2 场景 2 搜索策略

```python
async def search_cross_merchant(merchant_ids, query_vec):
    # 分离 SVIP 和普通商家
    svip_ids = [m for m in merchant_ids if is_svip(m)]
    normal_ids = [m for m in merchant_ids if not is_svip(m)]

    total_normal = sum(get_count(m) for m in normal_ids)

    tasks = []

    # 普通商家: 按总量选策略
    if normal_ids:
        if total_normal < 1_000_000:  # < 100 万
            tasks.append(scalar_brute_force(normal_ids, query_vec))
        else:
            tasks.append(hnsw_filtered("month", normal_ids, query_vec))

    # SVIP: 各自在 SVIP 分区内 HNSW filtered
    for sid in svip_ids:
        tasks.append(hnsw_filtered("svip", sid, query_vec))

    results = await asyncio.gather(*tasks)
    return merge_and_rank(results)
```

## 七、月分区的生命周期管理

两套分区 (标准 + SVIP) 采用完全相同的管理策略:

| 操作 | 频率 | 标准分区 | SVIP 分区 |
|------|------|---------|----------|
| 新建分区 | 每月 | month_YYYYMM | svip_YYYYMM |
| 冷热切换 | 每月 | 第 6→7 月 HNSW→DiskANN | 同 |
| TTL 清理 | 每月 | drop 过期月分区 | drop 过期月分区 |
| 索引构建 | 每月 | 新分区 build index | 同 |
| 内存管理 | 每月 | 热区 load, 冷区 release | 同 |

**一套 cron-scheduler 逻辑，操作两组分区名前缀 (month_ / svip_)。**

## 八、总结

### 8.1 架构决策

| 决策 | 方案 | 原因 |
|------|------|------|
| 数据分区 | 标准月分区 19 + SVIP 月分区 19 = **38 个分区** | 简单统一, 同一套生命周期管理 |
| 普通商家搜索 (99%+) | 标量查询 + 暴力搜索 | 精度 100%, < 10ms |
| SVIP 搜索 (<1%) | HNSW filtered ANN (选择率 ~6.25%) | 30~50ms |
| SVIP 阈值 | 商家图片量 ≥ 1000 万 | 每 SVIP 分区容纳 ~16 个 SVIP |
| SVIP 路由 | PG 路由表 svip_partition_roster | 唯一真相, 同月同商家只在一种分区 |
| 晋升/降级 | 下月生效, 不迁移历史数据 | 路由表修改即可, 历史数据自然过期 drop |
| TTL 清理 | drop 月分区 | 两套分区统一操作, 无 delete 问题 |
| 商家计数 | Redis 缓存, 写入时维护 | 月度评估晋升/降级阈值 |
| 向量缓存 | L1 进程 / L2 Redis | 减少普通商家 Milvus 数据传输 |

### 8.2 待验证事项

| 事项 | 方法 | 影响 |
|------|------|------|
| Milvus 低选择率自动退化 | 构造测试数据实测 | 若可行则普通商家也可直接用 Milvus search |
| 10 万条暴力搜索延迟 | 基准测试 | 确认普通商家暴力搜索边界 |
| SVIP 阈值 1000 万是否合理 | 统计实际商家分布 | 确认 SVIP 数量和分区容量 |
