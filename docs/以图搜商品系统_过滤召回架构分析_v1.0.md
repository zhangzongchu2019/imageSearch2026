# 以图搜商品系统 — 过滤与召回架构分析

> 版本: v1.0
> 日期: 2026-03-31
> 状态: 分析结论

---

## 一、问题背景

系统需要在 30 亿向量中按 merchant_id 过滤搜索。单商家典型图片量 ~1 万张，占全局 0.00025%。需要分析在不同过滤选择率下，Milvus 检索方案的可行性，以及对系统架构的影响。

---

## 二、核心矛盾

### 2.1 商家规模分布

| 商家类型 | 图片量 | 占全局 30 亿 | 占比商家数 |
|---------|--------|------------|----------|
| 典型商家 | ~1 万 | 0.00025% | 99%+ |
| 中等商家 | ~10 万 | 0.003% | 少量 |
| 大商家 | ~50 万 | 0.017% | 极少 |
| 头部商家 | ~800 万 | 0.27% | 极少数 |

### 2.2 HNSW filtered ANN 的局限

Milvus 的 HNSW filtered ANN 在图遍历过程中检查标量条件。当选择率极低时：

```
HNSW 遍历节点 → 检查 merchant_id → 不匹配 → 继续遍历
```

- 选择率 0.00025% (典型商家): 需遍历约 400 万节点才能找到 ~10 条匹配, **不可行**
- 选择率 > 5%: HNSW 正常高效工作

**小商家暴力搜索够快但 HNSW 过滤不行，大商家 HNSW 过滤能用但暴力搜索不行。**

## 三、分区方案分析

### 3.1 仅按月分区 (18 分区)

```
18 个月 + 常青 = 19 个分区
每月: 30 亿 / 18 ≈ 1.67 亿/分区
```

| 商家类型 | 每月图片 | 分区内选择率 | HNSW 可行？ |
|---------|---------|------------|-----------|
| ~1 万 | 556 | 0.00033% | 不可行 |
| ~800 万 | 44 万 | 0.27% | 边界 |

**结论**: 月分区对数据生命周期管理有价值 (冷热分离、TTL 清理)，但不解决 merchant_id 过滤问题。

### 3.2 月分区 × 哈希子分区

| 子分区数 | 总分区数 | 每子分区大小 | 典型商家选择率 | 头部商家选择率 |
|---------|---------|------------|-------------|-------------|
| 16 | 289 | 1044 万 | 0.005% | **4.3%** |
| 64 | 1,153 | 261 万 | 0.021% | 17% |
| 128 | 2,305 | 130 万 | 0.043% | 34% |
| 256 | 4,609 | — | — | **超 Milvus 4096 上限** |

**月 × 16 哈希结论**: 头部商家选择率 4.3% 可以让 HNSW 正常工作；典型商家 0.005% 仍不可行。

### 3.3 为什么纯分区方案走不通

要让 HNSW filtered ANN 对典型商家 (556 条/月) 高效工作，选择率需 > 5%:

```
每分区最大容量 = 556 / 5% = 11,120 条
需要分区数 = 1.67 亿 / 11,120 ≈ 15,000 个/月
总分区数 = 18 × 15,000 = 270,000 个 → 远超 Milvus 上限
```

## 四、最终方案：月分区 × 16 哈希 + 自适应搜索策略

### 4.1 数据分区

```
Milvus Collection: global_images
├── 202601_h00 ~ 202601_h15  (热区, HNSW)
├── 202602_h00 ~ 202602_h15  (热区, HNSW)
├── ...
├── 202606_h00 ~ 202606_h15  (热区, HNSW)   ← 6 个月 × 16 = 96 分区
├── 202507_h00 ~ 202507_h15  (冷区, DiskANN)
├── ...
├── 202412_h00 ~ 202412_h15  (冷区, DiskANN) ← 12 个月 × 16 = 192 分区
└── evergreen                                 ← 1 分区
                                              合计: 289 分区

标量索引: merchant_id (INVERTED), category_l1 (INVERTED), tags (INVERTED)
```

### 4.2 自适应搜索策略

根据商家图片总量选择不同搜索路径:

```
                    商家图片总量
                        │
           ┌────────────┼────────────┐
           ▼            ▼            ▼
       < 10 万      10万~100万     > 100 万
      (总量<10万)   (总量10~100万)  (总量>100万)
       99%+ 商家     少量商家       极少数头部
           │            │            │
           ▼            ▼            ▼
      标量查询       HNSW filtered   HNSW filtered
      + 暴力搜索     (子分区内)       (子分区内)
           │        选择率0.5~5%    选择率>4%
           ▼            │            │
        < 10ms       20~80ms       50~150ms

       精度100%      精度~99%       精度~98%
```

### 4.3 各规模商家的搜索表现

| 商家规模 | 搜索策略 | 子分区选择率 | 延迟 | 精度 |
|---------|---------|------------|------|------|
| ~1 万 (典型) | 标量查询 + 暴力搜索 | — | < 10ms | 100% |
| ~10 万 | 标量查询 + 暴力搜索 | — | < 50ms | 100% |
| ~50 万 | HNSW filtered | 0.27% | ~50ms | ~99% |
| ~800 万 (头部) | HNSW filtered | 4.3% | ~80ms | ~98% |

### 4.4 策略选择逻辑

```python
BRUTE_FORCE_THRESHOLD = 100_000  # 10 万

async def search_single_merchant(merchant_id, query_vec):
    image_count = await get_merchant_image_count(merchant_id)  # Redis 缓存

    if image_count < BRUTE_FORCE_THRESHOLD:
        # 小商家: 标量查询 + 暴力搜索 (精度 100%)
        return await scalar_query_brute_force(merchant_id, query_vec)
    else:
        # 大商家: HNSW filtered ANN (子分区内选择率足够)
        partition = get_hash_partition(merchant_id)
        return await hnsw_filtered_search(partition, merchant_id, query_vec)
```

### 4.5 商家图片计数维护

- 写入时: 原子递增 Redis 计数 `merchant_img_count:{merchant_id}`
- TTL 过期清理时: 批量递减
- 搜索时: 读取 Redis 计数决定策略
- 精度要求: 近似值即可，不影响功能正确性，仅影响策略选择

## 五、暴力搜索的数据传输优化

### 5.1 问题

标量查询 + 暴力搜索需要从 Milvus 传输向量数据:

```
1 万条 × 256d × 4 bytes = ~10 MB (全局向量)
1 万条 × 3 子图 × 128d × 4 bytes = ~15 MB (子图向量)
单次查询传输: ~25 MB
```

### 5.2 缓存方案

商家图搜开单是高频重复场景，同一商家一天搜几十上百次:

```
L1 进程内缓存: 热门 500 家 → ~12.5 GB
L2 Redis 缓存: 活跃 5000 家 → ~125 GB (Redis 集群)
L3 Milvus 标量查询: 缓存未命中时回源

缓存 TTL: 5 分钟 (写入可见性 ≤ 3 分钟, 留裕量)
```

### 5.3 Milvus 自动退化验证

Milvus 2.5 在过滤后候选数量极少时 (< 总量 ~0.1%)，可能自动退化为 brute-force。如果验证可行，则无需传输向量到进程内:

```python
# 验证测试: Milvus filtered search 在极低选择率下的表现
results = collection.search(
    data=[query_vec],
    anns_field="global_vec",
    param={"metric_type": "COSINE", "params": {"ef": 256}},
    expr="merchant_id == 'TEST_MERCHANT'",
    limit=100,
)
# 如果延迟 < 50ms 且召回完整 → 直接用 Milvus, 架构最简
# 如果延迟不可接受 → 启用缓存方案
```

|-----------|--------|---------------------|------|

### 6.2 场景 2 同样适用自适应策略

```python
async def search_cross_merchant(merchant_ids, query_vec):
    total_images = sum(await get_merchant_image_count(m) for m in merchant_ids)

    if total_images < BRUTE_FORCE_THRESHOLD:  # 10 万
        return await scalar_query_brute_force_multi(merchant_ids, query_vec)
    else:
        return await hnsw_filtered_search_multi(merchant_ids, query_vec)
```

## 七、月分区的独立价值

月分区虽不解决 merchant_id 过滤问题，但对数据管理至关重要:

| 价值 | 说明 |
|------|------|
| 冷热分离 | 近 6 月热区 HNSW (内存), 其余冷区 DiskANN (磁盘) |
| TTL 清理 | 直接 drop 过期月分区，无需逐条删除 |
| 滚动迁移 | 第 6 月 → 第 7 月，分区从热区降为冷区 |
| 索引独立 | 每月分区独立 build index |
| 加载灵活 | 热区分区常驻内存，冷区按需加载 |
| 上新隔离 | 新数据写入当月分区，不影响历史分区索引 |

## 八、总结

### 8.1 架构决策

| 决策 | 方案 | 原因 |
|------|------|------|
| 数据分区 | 月分区 × 16 哈希子分区 (289 分区) | 生命周期管理 + 大商家选择率提升 |
| 小商家搜索 (99%) | 标量查询 + 暴力搜索 | 精度 100%, < 10ms |
| 大商家搜索 (< 1%) | HNSW filtered ANN | 子分区选择率 > 4%, 可行 |
| 切换阈值 | 商家图片量 10 万条 | 暴力搜索性能边界 |
| 商家计数 | Redis 缓存, 写入时维护 | 搜索时决定策略 |
| 向量缓存 | L1 进程 / L2 Redis | 减少 Milvus 数据传输 |

### 8.2 待验证事项

| 事项 | 方法 | 影响 |
|------|------|------|
| Milvus 低选择率自动退化 | 构造测试数据实测 | 若可行则无需外置缓存 |
| 10 万条暴力搜索延迟 | 基准测试 | 确认阈值合理性 |
| 子分区 16 vs 32 | 对比测试 | 选择率 / 分区数 / 管理复杂度权衡 |
