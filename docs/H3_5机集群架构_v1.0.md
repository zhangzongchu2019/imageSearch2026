## 1. 数据规模与分片

### 30 亿数据 / 5 机

| 项 | 值 |
|----|----|
| 总图片 | 30 亿 |
| 每机分片 | **6 亿 / 机** |
| 单图存储 (1024d SQ8) | ~1.3 KB |
| 每机数据量 | **~780 GB** |
| 每机 RAM 占用 | **~1 TB** (约 50%) |

**关键优势**: 5 机集群让 RAM 从单机 2.7 TB 不够 → 5 机每台只用 1 TB, **可以用更高精度配置 (1024d 无 PCA)**。

### 单机详细 RAM 预算 (每机 6 亿)

```
组件                                  存储/图    6亿合计
─────────────────────────────────────────────────────
L0 pHash + dHash                     16 B       10 GB
L1 主索引 DINOv2 1024d SQ8           1024 B     614 GB
L1 HNSW 邻接表 (M=48)                230 B      138 GB
L2 热数据精排 fp16 (近1月 ~0.5亿)    2 KB       100 GB
标量字段 (pk, brand, category, etc.) 50 B       30 GB
RoaringBitmap 索引                   ~30 B      18 GB
─────────────────────────────────────────────────────
合计                                            ~910 GB / 机

RAM 余量: 2 TB - 910 GB = 1.1 TB (55% 闲置, 用于 OS/缓存/临时)
```

## 2. 分片策略

### Strategy: image_pk Hash 分片

```python
machine_idx = hash(image_pk) % 5
```

**优点**:
- 数据均匀分布
- 写入负载均衡
- 简单, 无热点

**缺点**:
- 查询需要扇出到所有 5 机 (federated search)
- 增加协调开销 (~10ms)

### 替代方案: 时间分片 + 副本

```
img_202601~02_xxx → 机 0
img_202603~04_xxx → 机 1
...
热数据 (近1月) → 全部 5 机副本 (查询本地)
```

**优点**:
- 时间过滤天然分区
- 热数据本地查询 (零跨机延迟)

**缺点**:
- 老月份机器负载低
- 副本同步复杂

### 最终选择: **混合分片** (推荐)

```
冷数据 (3 个月以上, 28 亿):
  → image_pk Hash 分片到 5 机
  → 每机存 5.6 亿冷数据

热数据 (近 3 个月, 2 亿):
  → 全副本到 5 机 (每机 0.4 亿热数据)
  → 查询本地, 零跨机延迟

最热数据 (近 1 周, ~1500万):
  → 内存常驻 fp16 精排向量
  → 每机 300 万 × 2KB = 6 GB
```

**单机存储 (混合分片)**:
```
冷数据 (5.6 亿):  ~720 GB
热数据 (0.4 亿):  ~52 GB
最热数据 (300万):  6 GB (额外 fp16)
合计:             ~780 GB / 机
```

## 3. Milvus 部署拓扑

### 3.1 组件分布 (基于 Milvus 2.5 分布式)

```
Machine 0 (Master):
  - etcd (cluster member 1)
  - rootcoord
  - mixcoord (datacoord+querycoord+indexcoord)
  - proxy (查询入口 1)
  - querynode-1
  - datanode-1
  - indexnode-1
  - kafka-broker-1
  - minio-1

Machine 1:
  - etcd (cluster member 2)
  - proxy (查询入口 2, 负载均衡)
  - querynode-2
  - datanode-2
  - indexnode-2
  - kafka-broker-2
  - minio-2

Machine 2:
  - etcd (cluster member 3)
  - querynode-3
  - datanode-3
  - indexnode-3
  - kafka-broker-3
  - minio-3

Machine 3:
  - querynode-4
  - datanode-4
  - indexnode-4
  - minio-4

Machine 4:
  - querynode-5
  - datanode-5
  - indexnode-5
  - minio-5
```

### 3.2 关键组件配置

| 组件 | 数量 | 备注 |
|------|------|------|
| etcd | 3 | 跨 3 机, 高可用 |
| proxy | 2 | 查询入口, 负载均衡 |
| rootcoord | 1 | (单点, 待 Milvus 3.x 支持 HA) |
| mixcoord | 1 | 整合 data/query/index coord |
| querynode | **5** | 每机 1 个, 各持有 1/5 数据 |
| datanode | **5** | 写入分片 |
| indexnode | **5** | 索引构建分片 |
| kafka | 3 | MQ_TYPE=kafka 集群 |
| minio | 5 | 对象存储 (向量数据) |

### 3.3 查询数据流

```
用户请求
  ↓
LB (nginx/haproxy)
  ↓
Proxy (Machine 0/1)
  ↓
QueryCoord 协调
  ↓
扇出到 5 个 querynode (并行)
  ├── querynode-1 (机0): 搜本地 1/5 数据 → Top 100
  ├── querynode-2 (机1): 搜本地 1/5 数据 → Top 100
  ├── querynode-3 (机2): 搜本地 1/5 数据 → Top 100
  ├── querynode-4 (机3): 搜本地 1/5 数据 → Top 100
  └── querynode-5 (机4): 搜本地 1/5 数据 → Top 100
  ↓
Proxy 合并 Top 100 × 5 = 500 个候选
  ↓
全局 Top 10 排序 → 返回
```

**预期延迟分解**:
```
Network LB → Proxy:           1 ms
Proxy → QueryCoord:           1 ms
扇出 5 querynode (并行):      20 ms (单 querynode 内部完成 HNSW_SQ8 + refine)
合并 500 → Top 10:            2 ms
Proxy → 用户:                 1 ms
─────────────────────────────────────
合计:                         25 ms (热查询)
                              35 ms (冷查询, 含 SSD I/O)
```

## 4. 网络架构

### 4.1 网络拓扑

```
┌────────────────────────────────────────────────┐
│         External LB (nginx)                     │
│         - VIP, 接收用户查询                      │
└─────────────────┬──────────────────────────────┘
                  │ HTTP/gRPC
                  ↓
┌────────────────────────────────────────────────┐
│  内部 25Gbps 网络 (机器间)                       │
│                                                 │
│  M0 ━━━━━━ M1 ━━━━━━ M2 ━━━━━━ M3 ━━━━━━ M4    │
│   │         │         │         │         │     │
│   └─────────┴─────────┴─────────┴─────────┘     │
│              Mesh / Star 拓扑                    │
└────────────────────────────────────────────────┘
```

### 4.2 网络流量估算

**单查询的跨机流量**:
```
扇出: Proxy → 5 个 querynode 各发 1 个 query (1024d float32)
  Query payload: 4 KB × 5 = 20 KB
  
回收: 5 个 querynode → Proxy 各返 Top 100 (image_pk + score)
  Per result: 64 + 4 = 68 B
  Per response: 68 × 100 = 6.8 KB
  Total: 6.8 KB × 5 = 34 KB

每查询双向: ~54 KB
```

**1000 QPS 时**:
```
54 KB × 1000 / s = 54 MB/s = 432 Mbps (双向)
远小于 25 Gbps 内网带宽 (约 2%)
```

**结论**: 网络不是瓶颈, 25 Gbps 内网完全足够。

### 4.3 高可用网络设计

| 组件 | HA 策略 |
|------|--------|
| 外部 LB | nginx + keepalived (2 机), VIP 漂移 |
| Proxy | 2 副本 (Machine 0/1), LB 自动剔除 |
| etcd | 3 副本 (M0/M1/M2), 多数派写入 |
| QueryNode | 5 副本 (每机 1), Milvus 自动重分配 |
| RootCoord | 1 副本 (M0), Milvus 2.5 暂未支持 HA, 故障切换 ~30s |

## 6. 推理服务部署

### 6.1 DINOv2 推理服务

每机部署 1 个 inference service, 复用本机 4 × 4090D:
```
machine X:
  inference-service-X (FastAPI, 端口 8090)
    - DINOv2-L on GPU 0
    - DINOv2-L on GPU 1
    - DINOv2-L on GPU 2
    - DINOv2-L on GPU 3
    - 内部 batching 队列
```

**总推理能力**:
```
单机: 4 GPU × 500 img/s = 2000 img/s
集群: 5 机 × 2000 = 10,000 img/s
```

**对比业务需求**:
- 在线查询: ~100 QPS → 100 img/s, 完全 cover
- 在线写入: ~10 img/s → 100ms 即可
- 批量重推理: 30 亿 / 10000 = 83 小时 (一次性任务)

### 6.2 搜索服务

每机部署 1 个 H3 search service:
```
machine X:
  h3-search-service-X (FastAPI, 端口 9810)
    - 接收 image base64 + 过滤条件
    - 调用本机 inference-service 提取特征
    - 调用 Milvus proxy 检索
    - 返回 Top 10
```

LB 在前面分发请求, 5 机均衡负载。

## 7. 灰度迁移路线

### 7.1 迁移阶段

```
Phase A: 单机验证 (本机, 进行中)
  - 100 万测试图 + Milvus standalone (现状)
  - 验证 H3 流程跑通

Phase B: 单机生产模拟
  - 在本机部署完整 H3 服务
  - 用现有 1.5 亿数据测试
  - 双写双读对比新旧方案
  
Phase C: 1 机正式部署
  - 把本机升级为生产配置
  - 单机生产环境上线
  - 灰度 1% 流量

Phase D: 5 机集群扩容
  - 加入 4 台新机器
  - Milvus 重新分片 (重启写入)
  - 灰度 10% → 50% → 100%

Phase E: 旧服务下线
  - 流量 100% 切到新集群
  - 旧 CLIP Collection 下线
  - 资源回收
```

### 7.2 迁移工具

| 工具 | 用途 |
|------|------|
| Milvus Backup | 集合 snapshot + 跨集群迁移 |
| 双写脚本 | 同时写入新旧 Collection, 验证一致 |
| 双读对比 | 同 query 跑新旧两套, 对比结果 |
| 流量影子 | 复制生产流量到新集群, 不返回结果 |

## 9. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 单机故障 | 中 | 数据 1/5 不可用 | 热数据全副本 + 冷数据按 hash 重分配 (Milvus 自动) |
| 网络分区 | 低 | etcd 脑裂 | 3 节点多数派 + 跨机房部署 |
| RootCoord 单点 | 中 | 短暂不可写 | Milvus 2.5 限制, 故障转移 30s, 等 3.x |
| 索引构建 OOM | 低 | 索引中断 | 监控 RAM, 控制并发构建数 |
| 跨机查询超时 | 低 | P99 > 100ms | 设置 query timeout, 部分 querynode 失败时返回部分结果 |
| Milvus 升级 | 中 | 服务中断 | 滚动升级, querynode 一台一台来 |
