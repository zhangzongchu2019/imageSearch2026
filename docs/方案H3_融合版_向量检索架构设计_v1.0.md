## 2. 方案 H3 架构

### 2.1 总览

```
┌─────────────────────────────────────────────────────────┐
│  L0: 指纹去重                                             │
│    技术: pHash + dHash (Milvus BIN_FLAT + Hamming)      │
│    存储: 16 B/图                                         │
│    延迟: ~1 ms                                           │
│    场景: 完全重复, 近重复 (压缩/轻度裁剪)                  │
└─────────────────────────────────────────────────────────┘
                          ↓ 未命中
┌─────────────────────────────────────────────────────────┐
│  L1: 主召回 (DINOv2 + HNSW_SQ8 + RoaringBitmap 过滤)     │
│    模型: DINOv2-L 预训练 → PCA 512d (零训练)             │
│    索引: Milvus HNSW_SQ8 + refine=FP16 (内置精排)        │
│    过滤: brand/category/merchant_id 算子下推              │
│    存储: 512 B (SQ8) + 180 B (HNSW 邻接) = ~700 B        │
│    延迟: ~20 ms                                          │
└─────────────────────────────────────────────────────────┘
                          ↓ Top 100
┌─────────────────────────────────────────────────────────┐
│  L2: 热数据精排                                           │
│    对象: 近 3 个月热数据 (~2 亿)                          │
│    存储: DINOv2 原始 1024d fp16, 内存常驻                │
│    计算: Top 100 → 精确余弦 → Top 20                     │
│    存储成本: 2KB × 2亿 = 400 GB                          │
│    延迟: ~5 ms                                           │
└─────────────────────────────────────────────────────────┘
                          ↓ Top 20
┌─────────────────────────────────────────────────────────┐
│  L3: 冷数据层 (28 亿)                                     │
│    索引: DiskANN (向量在 SSD, 图结构在 RAM)               │
│    触发: 用户指定历史数据或全量搜索                        │
│    延迟: 30-50 ms                                        │
└─────────────────────────────────────────────────────────┘
                          ↓
                  Top 10 返回结果

总延迟预算:
  热查询 (L0 + L1 + L2):        25-30 ms  ← 90% 场景
  冷查询 (L0 + L1 + L3):        40-60 ms  ← 10% 场景
  局部场景 (+ optional patch):  +20 ms    ← 未来扩展
```

### 2.2 设计原则

1. **零训练成本**: 使用 DINOv2 预训练权重 + 后置 PCA, 不做 fine-tune / QAT
2. **两级向量存储**: SQ8 主索引 (压缩) + FP16 复排向量 (精度)
3. **多通道检索**: 指纹走指纹索引, 向量走向量索引, 过滤走 Bitmap
4. **冷热分层**: 热数据全内存 HNSW, 冷数据 SSD DiskANN
5. **工程可预测**: 每一步延迟 <10ms, 总延迟可预估
6. **降级可控**: 可按资源选择 256d/384d/512d 维度

## 3. 关键技术决策

### 3.1 模型选型

| 候选 | 维度 | 优势 | 劣势 | 决策 |
|------|------|------|------|------|
| CLIP ViT-L-14 (现有) | 768 | 零样本分类强, 已有数据 | **截图场景弱** | ❌ 不足 |
| SSCD ResNet50 | 512 | **副本检测 SOTA** | 无语义理解 | ⚠️ 辅助 |
| **DINOv2-L** | **1024** | **副本+语义兼顾** | 需降维 | ✅ **主力** |
| SigLIP 2 | 768 | 最新, 语义+定位 | Milvus 生态不成熟 | ⏭️ 未来 |
| EVA-CLIP | 768 | CLIP 增强 | 副本检测仍偏弱 | ❌ |

**理由**: DINOv2 自监督训练天然对裁剪/截图鲁棒 (同 SSCD), 同时保留了语义相似搜索能力 (类似 CLIP)。是唯一"单模型多场景"的候选。

### 3.2 维度选择

| 维度 | 单图 (SQ8) | 30亿存储 | 精度保留 | RAM 总需 |
|------|-----------|---------|---------|---------|
| 256d | 256 B | 768 GB | 92% | 1.8 TB ✅ |
| **384d** | **384 B** | **1.15 TB** | **95%** | **2.0 TB ⚠️** |
| 512d | 512 B | 1.54 TB | 97% | 2.3 TB ❌ |
| 768d | 768 B | 2.3 TB | 99% | 3.0 TB ❌ |

**初始决策**: **384d SQ8** (精度 / 内存的最佳平衡)
**备用**: 若 RAM 吃紧, 降到 256d (精度损失仅 3%)

### 3.3 索引选择

| 索引 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| IVF-PQ | 压缩比高 | **亿级误差累积严重** | ❌ |
| HNSW (FP32) | 召回最高 | 存储大 | ❌ |
| **HNSW_SQ8 + refine FP16** | **平衡精度与存储, Milvus 原生** | 略慢于 FP32 | ✅ **主索引** |
| HNSW_PQ | 存储最小 | 精度下降大 | ❌ |
| ScaNN | Google 生产级 | Milvus 支持不成熟 | ❌ |
| **DiskANN** | **SSD 支持十亿级** | RAM 吃 ~40% | ✅ **冷层** |

**主索引**: `HNSW_SQ8 + refine=FP16, M=32, efConstruction=500`
**冷层**: `DiskANN, R=48, L_build=100, L_search=80`

### 3.4 量化策略

| 方案 | 单图 | 精度损失 | 决策 |
|------|------|---------|------|
| FP32 原始 | 1536 B | 0% | ❌ 存储过大 |
| FP16 | 768 B | <1% | ⚠️ 仅用于 L2 热数据 |
| **SQ8 + refine FP16** | **384 B (主) + 768 B (副)** | **~2%** | ✅ **L1 主索引** |
| PQ (24B) | 24 B | 10-15% | ❌ 亿级失真 |

**关键**: Milvus 2.5 的 `HNSW_SQ` 索引支持 `refine` 参数, 查询时内部做"SQ8 剪枝 + FP16 精排", 是官方推荐的两级存储实现。

### 3.5 过滤策略

**场景**: 商品图关联多个商家 ID, 查询时需要 "属于商家 A/B/C 之一" 的过滤。

| 方案 | 实现 | 问题 |
|------|------|------|
| 先 ANN 后过滤 | 检索 Top 1000 → 过滤 | **召回崩塌**, Top 1000 可能无符合商家 |
| 先过滤后搜 | 查询前 bitmap 交集 → 扫描 | 大商家集合时慢 |
| **算子下推 (操作符融合)** | **HNSW 遍历时实时 bitmap.test()** | ✅ **最优** |

**技术栈**: `RoaringBitmap` (由 Daniel Lemire 原版实现, Milvus 2.5 内置支持)

```python
# 过滤实现
merchant_bitmap = RoaringBitmap([img_ids_belonging_to_merchant_123])
search_expr = f"merchant_id in [123, 456, 789]"
# Milvus 自动下推到 HNSW 遍历时的节点过滤
```

**性能**: 单次 bitmap.test() 在 AVX-512 下 1-3 ns, HNSW 遍历 ~1000 节点, 总过滤开销 <5ms。

## 4. 存储预算

### 4.1 单图字节级预算

```
┌───────────────────────────────────────┐
│  L0 指纹                                │
│    pHash            8 B                 │
│    dHash            8 B                 │
│  L0 小计           16 B                 │
├───────────────────────────────────────┤
│  L1 主索引 (HNSW_SQ8)                   │
│    SQ8 向量       384 B (384d)          │
│    HNSW 邻接表 摊销 ~150 B              │
│  L1 主小计        534 B                 │
├───────────────────────────────────────┤
│  L1 复排向量 (FP16, 可选)                │
│    384d fp16      768 B                 │
│    (热数据常驻, 冷数据从 SSD 加载)       │
├───────────────────────────────────────┤
│  标量字段                                │
│    image_pk         8 B                 │
│    url_hash         8 B                 │
│    ts_month         4 B                 │
│    category_l1      4 B                 │
│    category_l2      4 B                 │
│    brand_id         4 B                 │
│    merchant_set_id  8 B                 │
│    flags            4 B                 │
│  标量小计          44 B                 │
├───────────────────────────────────────┤
│  Bitmap 索引摊销 (按 30 亿估算)           │
│    brand Bitmap 摊销 ~10 B              │
│    merchant Bitmap 摊销 ~15 B           │
│  Bitmap 小计       25 B                 │
└───────────────────────────────────────┘

单图合计:
  最小版 (仅 L0 + L1 主):          ~620 B
  热数据完整版 (含 L1 复排):       ~1380 B
  冷数据版 (L1 复排走 SSD):        ~620 B
```

### 4.2 30 亿总存储

| 组件 | 存储 | 介质 |
|------|------|------|
| **RAM 常驻** | | |
| L0 指纹索引 | 48 GB | RAM |
| L1 主索引 (HNSW_SQ8) | 1.2 TB | RAM |
| L1 HNSW 邻接表 | 450 GB | RAM |
| L2 热数据 FP16 (2亿) | 300 GB | RAM |
| 标量字段 + Bitmap | 200 GB | RAM |
| **RAM 合计** | **~2.2 TB** | ⚠️ 需缩减 |
| **SSD 存储** | | |
| L3 冷数据 DiskANN (28亿) | ~4 TB | SSD |
| DiskANN 索引 | ~500 GB | SSD |
| **SSD 合计** | **~4.5 TB** | ✅ |

### 4.3 RAM 缩减方案

**目标**: RAM ≤ 2 TB (服务器上限)

| 选项 | 节省 | 代价 |
|------|------|------|
| L1 降到 256d | 400 GB | 精度 95% → 92% |
| 热数据从 2亿 → 1亿 | 150 GB | L2 覆盖面↓ |
| HNSW M=32 → M=24 | 150 GB | 召回率 ↓ 1% |

**推荐缩减路径**: 256d + 1亿热数据 + M=24 → **~1.7 TB RAM**, 留 300 GB 余量。

## 5. 延迟预算

### 5.1 热查询链路 (90% 场景)

```
阶段                         预算     说明
────────────────────────────────────────────
查询特征提取 (GPU)            4 ms    DINOv2-L 推理 (224×224)
L0 指纹查询 (Milvus BIN)      1 ms    Hamming 距离 Top 5
L1 HNSW_SQ8 召回             15 ms    ef_search=200, Top 100
  - SQ8 剪枝                  8 ms
  - FP16 refine               5 ms
  - Bitmap 过滤开销           2 ms
L2 热数据精排                  5 ms    Top 100 → Top 20
  - FP16 向量加载              1 ms
  - 余弦相似度 (GPU/SIMD)      3 ms
  - 排序                      1 ms
结果封装返回                   1 ms
────────────────────────────────────────────
合计                        26 ms
```

L3 DiskANN 精排              30 ms    SSD I/O 为主

────────────────────────────────────────────

## 6. Milvus Schema 设计

### 6.1 主 Collection Schema

```python
from pymilvus import FieldSchema, CollectionSchema, DataType

fields = [
    # 主键
    FieldSchema(name="image_pk", dtype=DataType.VARCHAR, max_length=64, is_primary=True),

    # L0 指纹 (Binary)
    FieldSchema(name="phash", dtype=DataType.BINARY_VECTOR, dim=64),
    FieldSchema(name="dhash", dtype=DataType.BINARY_VECTOR, dim=64),

    # L1 主向量 (HNSW_SQ8)
    FieldSchema(name="global_vec", dtype=DataType.FLOAT_VECTOR, dim=384),

    # L2 复排向量 (仅热数据需要, 可选 nullable)
    # 可以放在独立 Collection 或用属性字段承载

    # 标量字段
    FieldSchema(name="url_hash", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="ts_month", dtype=DataType.INT32),
    FieldSchema(name="category_l1", dtype=DataType.INT32),
    FieldSchema(name="category_l2", dtype=DataType.INT32),
    FieldSchema(name="brand_id", dtype=DataType.INT32),
    FieldSchema(name="merchant_ids", dtype=DataType.ARRAY, element_type=DataType.INT32, max_capacity=100),
    FieldSchema(name="created_at", dtype=DataType.INT64),
]
```

### 6.2 索引配置

```python
# L0: pHash/dHash 二进制索引
phash_index = {
    "index_type": "BIN_FLAT",  # 或 BIN_IVF_FLAT (大规模)
    "metric_type": "HAMMING",
}

# L1: 主向量索引
global_vec_index = {
    "index_type": "HNSW_SQ",
    "metric_type": "COSINE",
    "params": {
        "M": 32,                  # 邻居数
        "efConstruction": 500,    # 构建质量
        "sq_type": "SQ8",         # 8-bit 量化
        "refine": True,           # 启用精排
        "refine_type": "FP16",    # 精排用 fp16
    }
}

# 标量索引 (用于 Bitmap 过滤)
scalar_indexes = [
    ("brand_id", "STL_SORT"),
    ("category_l1", "STL_SORT"),
    ("merchant_ids", "BITMAP"),  # 多值字段用 BITMAP 索引
]
```

### 6.3 查询示例

```python
# 混合搜索 (向量 + 过滤)
results = collection.search(
    data=[query_vector],
    anns_field="global_vec",
    param={
        "metric_type": "COSINE",
        "params": {"ef": 200}
    },
    limit=100,
    expr="brand_id in [123, 456] and category_l1 == 1001 and 123 in merchant_ids",
    output_fields=["image_pk", "category_l1", "category_l2"]
)
```

## 9. 未被采纳的选项 (供后续讨论)

### 9.1 多头 DELG 架构 (ChatGPT 建议)

```
G-head (全局) + R-head (区域) + S-head (语义)
```

**不采纳理由**: R-head 需要定义和训练"显著区域提取器", 整体训练 7-10 周。

**未来可启用条件**: 当 Phase 2 的评测数据显示 DINOv2 单头对"图中图"场景召回率 <70% 时, 考虑引入 R-head。

### 9.2 QAT 量化感知训练 (Gemini 建议)

**不采纳理由**: 需要 2-3 周训练周期, 且 DINOv2 直接 PCA+SQ8 的精度损失已经可控。

**未来可启用条件**: 当 256d 以下维度精度不够, 但又无法扩 RAM 时, 通过 QAT 训练专用模型。

### 9.3 ScaNN 索引 (ChatGPT 建议)

**不采纳理由**: Milvus 2.5 的 ScaNN 支持仍在实验阶段, 文档少, 社区案例少。

**未来可启用条件**: Milvus 3.x 正式支持 ScaNN 后, 做 AB 测试验证是否比 HNSW_SQ8 快。

### 9.4 SuperPoint 局部特征重排 (Qwen 建议)

**不采纳理由**: SuperPoint 在电商白底图上几乎无特征点, 且 SuperPoint+SuperGlue 单对耗时 13-25 ms, Top 50 重排 = 650 ms。

**未来可启用条件**: 若有带纹理的商品搜索需求 (如艺术品、复古单品), 可考虑作为可选通道。

### 9.5 DINOv2 Patch 级匹配 (自己提出)

**暂缓理由**: Patch 级匹配最适合"图中图"场景, 但实现复杂度高 (需要 patch tokens 存储)。

**未来可启用条件**: Phase 2 发现图中图场景召回不足时, 在 L2 阶段添加 patch matching 作为可选 rerank。
