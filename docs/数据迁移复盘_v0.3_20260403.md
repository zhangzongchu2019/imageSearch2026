# 数据迁移复盘: v0.3 两千万数据导入

> 日期: 2026-04-02 ~ 2026-04-03 | 分支: isv3-0401

## 1. 目标

将以图搜商品系统从 ViT-B-32 (512→256) 升级到 ViT-L-14 (768→256) 两级分类架构:
- 清空旧数据 (~500 万条 ViT-B-32 向量)

## 3. 遇到的问题与解决

### 3.1 下载速度慢 (6 并发, 12 张/秒)

**原因**: COS 公网带宽限制, 每张原图 ~800KB, 单张下载 4-10s

**解决**: COS 服务端缩图 `?imageMogr2/thumbnail/224x224`
- 图片从 ~800KB → ~20KB (40 倍缩小)
- CLIP 输入就是 224×224, 不影响特征质量
- 速度提升 ~4 倍

**教训**: 始终使用 CDN/COS 的图片处理能力, 不要下载原图做本地 resize

### 3.2 多 GPU 并行失败

**尝试**:
1. `torch.multiprocessing.spawn` → CUDA context 死锁
2. 4 个独立 `extract_single_gpu.py` → CPU 预处理争抢, 1-3 img/s
3. 2 卡并行 (GPU0 先跑热, 再加 GPU1) → GPU1 30 img/s, 但 CPU 仍争抢

**根因**: PIL Image.open + preprocess 是 CPU 密集, 4 进程同时加载 10 万张小文件导致 I/O 争抢

**最终方案**: 单卡 78 img/s (跳过 FashionSigLIP L2/L3)

**教训**:
- 多 GPU 并行需要数据预加载 (DataLoader + num_workers)
- 或者每个进程处理自己的独立文件目录 (不共享 /tmp)
- FashionSigLIP 逐张推理是瓶颈, 已改为批量推理

### 3.3 衍生 OOM (cgroup 64GB)

**原因**: 1960 万条 × 256 float × 4 bytes + dict 开销 ≈ 80-100GB, 超过 cgroup 64GB 限制

**解决**: 分批衍生 (100 万/批), 每批写完释放内存

**教训**: 生产容器必须关注 cgroup 内存限制, 不能依赖宿主机 2TB 内存

### 3.4 Milvus etcd 崩溃 (核心问题, 4 次)

**崩溃链**: 大量 upsert → 产生大量 segment → 每 segment 11 个 indexing task → task scheduler 每秒轮询 etcd → etcd apply 延迟 25s → session lease 超时 → 全组件退出

**尝试**:
1. 降低 batch size (5000→2000) + 批间等 30s → 仍崩溃
2. 降低批量 (100 万→50 万) + 等 60s → 仍崩溃 (4 批后)
3. **Drop 索引** → 成功! 无 indexing task, etcd 零压力

**根因**: Milvus standalone 模式下 index node 只有 1 个 slot, 大量 task 排队每秒轮询 etcd 状态

**最终方案**:
```
写入前: drop 所有索引
写入中: 无 indexing task, etcd 无压力
写入后: 逐个重建索引
```

**同时优化 etcd 配置**:
- heartbeat 100ms → 500ms
- election timeout 1s → 5s
- session TTL 30s → 120s

**教训**:
- Milvus standalone 不适合带索引的大批量写入
- 生产环境应考虑分布式部署 (独立 indexNode)
- 批量导入前必须 drop 索引

### 3.5 etcd 数据丢失

**原因**: 手动 `docker rm` + `docker run` 重建 etcd 容器时, 新容器的 data-dir (`default.etcd`) 是空的。虽然挂载了 volume (`etcd-data:/etcd`), 但 etcd 写数据到容器内的 `default.etcd` 而非 volume 挂载点

**后果**: Milvus collection 元数据丢失, 需要从备份重新导入

**教训**:
- 不要随意重建 etcd 容器
- etcd data-dir 必须明确指定到 volume 挂载点: `--data-dir=/etcd`
- 或用 docker-compose 管理, 不手动 docker run

## 5. 代码优化 (已完成)

1. **FashionSigLIP 批量推理**: 从逐张 Image.open → forward 改为收集 batch → 一次 encode_image → 矩阵分类
2. **preprocess 快速路径**: 224×224 图片跳过 Resize+CenterCrop, 直接 ToTensor+Normalize
3. **Milvus 自动恢复**: derive 脚本带 3 次重试 + 自动 docker restart
4. **COS 缩图**: 下载时追加 `?imageMogr2/thumbnail/224x224`
5. **httpx 日志抑制**: 高并发下载时不打印每条 HTTP 请求日志
